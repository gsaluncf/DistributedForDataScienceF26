dbt Core: SQL-First Data Transformation

Transform data in your warehouse using plain SQL. Version-controlled, tested, documented, and portable across any data platform.

Part 1 What Is dbt Core?

1.1 The Problem It Solves

Before dbt, the typical data transformation workflow looked like this:

  • A pile of SQL scripts in a shared drive, numbered 01_clean.sql, 02_join.sql, etc.
  • No tests — if bad data slipped through, you found out from a broken dashboard
  • No documentation — only the original author knew what each script did
  • No dependency tracking — you ran scripts in order and hoped for the best
  • No version control — "final_v3_FINAL_USE_THIS.sql"

dbt fixes all of this by bringing software engineering practices — version control, testing, documentation, modularity, CI/CD — to SQL-based data transformation.

1.2 The "T" in ELT

dbt handles only the Transform step. It assumes raw data is already in the warehouse.

dbt does not extract or load data. It works entirely inside your warehouse, transforming raw data into clean, tested, documented tables using SQL SELECT statements.

1.3 How It Works (60-Second Version)

  1. You write SQL SELECT statements in .sql files called models
  2. Each model becomes a table or view in your warehouse
  3. Models reference each other with {{ ref('model_name') }} — dbt builds the DAG
  4. You define tests in YAML: unique, not_null, accepted_values
  5. Run dbt run to execute all models in dependency order
  6. Run dbt test to validate data quality
Key Insight

A dbt model is just a SQL SELECT statement. dbt wraps it in CREATE TABLE AS SELECT or CREATE VIEW AS for you. You never write DDL.

Part 2 Anatomy of a dbt Project

Every file has a clear role. Models are organized by data quality tier.

Part 3 Core Concepts

3.1 Models: SQL SELECT = Table

A dbt model is a .sql file containing a single SELECT statement. When you run dbt run, dbt compiles each model and executes it against your warehouse, creating a table or view.

-- models/staging/stg_holdings.sql
-- This is the ENTIRE file. Just a SELECT.

SELECT
    TRIM(REGEXP_REPLACE(Ticker, '^[A-Z]+:', '')) AS ticker,
    Company_Name AS company_name,
    CAST(REGEXP_REPLACE(Cost_Basis, '[$,]', '') AS DOUBLE) AS cost_basis,
    CAST(Shares_Held AS INT) AS shares_held,
    Sector AS sector
FROM {{ source('pipeline', 'bronze_holdings') }}
WHERE Ticker IS NOT NULL

dbt takes this SELECT and wraps it in CREATE VIEW AS (or CREATE TABLE AS depending on your materialization config). You never manually write DDL.

3.2 ref() and source(): Dependency Declaration

These two functions are the heart of dbt's DAG resolution — conceptually identical to dlt.read() in Lakeflow Declarative Pipelines:

Function Purpose DLT Equivalent
{{ source('group', 'table') }} Reference a raw table that exists outside dbt Reading from an external table
{{ ref('model') }} Reference another dbt model (declares dependency) dlt.read("table_name")

When dbt sees {{ ref('stg_holdings') }} inside portfolio_valuation.sql, it knows: (1) replace this with the actual table name, and (2) run stg_holdings before portfolio_valuation.

3.3 Materializations

Type What dbt Creates When to Use
view CREATE VIEW AS SELECT ... Staging models, lightweight transforms. Always fresh, but recomputes on every query.
table CREATE TABLE AS SELECT ... Mart/Gold models. Data is stored on disk. Fast to query, but stale until dbt run.
incremental INSERT INTO ... SELECT ... WHERE ... Large event tables. Append only new/changed rows instead of rebuilding the whole table.
ephemeral CTE (inlined into downstream queries) Helper logic that shouldn't be a real table. Like a private function.

3.4 Tests: Data Quality as Code

dbt tests are SQL queries that should return zero rows. If any rows are returned, the test fails. Tests are declared in schema.yml:

# schema.yml
models:
  - name: stg_holdings
    columns:
      - name: ticker
        tests:
          - unique          # No duplicate tickers
          - not_null        # Every row has a ticker
      - name: sector
        tests:
          - accepted_values:
              values: ['Technology', 'Consumer', 'Financials', 'Healthcare']

Behind the scenes, dbt compiles these to SQL. For example, unique on ticker becomes:

SELECT ticker
FROM workspace.dbt_demo.stg_holdings
GROUP BY ticker
HAVING COUNT(*) > 1
-- Returns rows only if there are duplicates → test fails
Compare: dbt Tests vs DLT Expectations

In DLT, @dlt.expect_or_drop() silently removes bad rows during pipeline execution. In dbt, tests run after the data is built — they validate the output rather than filtering the input. Both approaches have merit: DLT is proactive (prevent bad data), dbt is reactive (detect bad data).

Part 4 The Dependency Graph

dbt builds its DAG by scanning your models for ref() and source() calls. The exact same concept as DLT's two-phase approach (loading → execution), but using Jinja template resolution instead of Python decorators.

Same 4-level DAG as notebooks 5 and 6. dbt discovered it from {{ ref() }} calls in your .sql files.
Same Pattern, Different Mechanism

DLT resolves the DAG during Python module loading by inspecting dlt.read() calls. dbt resolves the DAG during Jinja template compilation by parsing {{ ref() }} calls. The result is identical: a directed acyclic graph that determines execution order.

Part 5 The Same Pipeline, Multiple Approaches

5.1 Silver Holdings: Two Syntaxes

DLT (Lakeflow)

@dlt.table(
    comment="Cleaned holdings",
    table_properties={"quality":"silver"}
)
@dlt.expect_or_drop(
    "valid_ticker",
    "Ticker IS NOT NULL")
def silver_holdings():
    return (dlt.read("bronze_holdings")
        .withColumn("Ticker",
            trim(regexp_replace(
                col("Ticker"),
                r"^[A-Z]+:", "")))
        .withColumn("Cost_Basis",
            regexp_replace(
                col("Cost_Basis"),
                r"[\$,]", "")
            .cast(DoubleType()))
        .dropDuplicates(["Ticker"])
    )

dbt Core

-- stg_holdings.sql
-- Just SQL. That's it.

SELECT
    TRIM(REGEXP_REPLACE(
        Ticker, '^[A-Z]+:', ''))
        AS ticker,
    CAST(REGEXP_REPLACE(
        Cost_Basis, '[$,]', '')
        AS DOUBLE) AS cost_basis,
    Sector AS sector
FROM {{ source('pipeline',
    'bronze_holdings') }}
WHERE Ticker IS NOT NULL
QUALIFY ROW_NUMBER() OVER (
    PARTITION BY ticker
    ORDER BY ticker) = 1

The transformation logic is the same. The key differences are the execution model and the surrounding infrastructure.

5.3 Other Approaches Worth Knowing

Tool What It Is Pros Cons
dbt Cloud Hosted dbt with browser IDE, scheduler, and CI/CD No infrastructure, team features, auto-docs Costs money, still SQL-only
Dataform Google's dbt-like tool (built into BigQuery) Free with BigQuery, SQLX syntax BigQuery-only, smaller community
Airflow DAG-based workflow orchestrator Handles any task (Python, SQL, APIs, ML) Complex setup, overkill for SQL-only
SQLMesh Open-source dbt alternative with virtual previews Column-level lineage, virtual environments Newer, smaller community
Stored Procs Database-native transformation scripts No external tools, familiar to DBAs No testing, no docs, no version control
Raw SQL Scripts Manual .sql files run in order Simple, no learning curve No DAG, no tests, no docs, fragile
When to Use What
  • dbt Core — Your org uses multiple warehouses, or you want open-source with CI/CD
  • DLT — You're all-in on Databricks and want platform-managed pipelines
  • Imperative PySpark — You need custom logic, ML, or complex Python
  • Raw SQL — Quick one-off analysis, no pipeline needed

Part 6 Getting Started on Databricks

6.1 Install dbt-databricks

The dbt-databricks adapter includes dbt-core as a dependency:

# In a Databricks notebook (first cell):
%pip install dbt-databricks

# On your local machine:
pip install dbt-databricks

6.2 Configure profiles.yml

You need your Databricks workspace URL, a personal access token, and a SQL Warehouse HTTP path. In the notebook, we generate this dynamically. On your local machine:

# ~/.dbt/profiles.yml
custodian_pipeline:
  target: dev
  outputs:
    dev:
      type: databricks
      host: dbc-xxxxxxxx-xxxx.cloud.databricks.com
      http_path: /sql/1.0/warehouses/xxxxxxxxxxxxxxxxx
      token: dapiXXXXXXXXXXXXXXXXXXXXXXXXXXXXXX
      catalog: workspace
      schema: dbt_demo
      threads: 4
Security

profiles.yml contains credentials and should never be committed to Git. In production, use environment variables or a secrets manager.

6.3 Run the Pipeline

# Parse and validate the project
dbt debug

# Execute all models in dependency order
dbt run

# Validate data quality
dbt test

# Generate documentation site
dbt docs generate
dbt docs serve

6.4 Hands-On Notebook

The nb7_dbt.py notebook walks you through the full setup step-by-step inside Databricks. It creates a mini dbt project, writes all the model files, and runs dbt run and dbt test against your SQL Warehouse.

Teaching Pattern

Running dbt inside a Databricks notebook is a teaching convenience, not a production pattern. In the real world, dbt runs from your terminal, CI/CD pipeline (GitHub Actions, GitLab CI), or a scheduler (Airflow, Dagster, dbt Cloud).

The Big Idea

dbt is what happens when the data world applies the lessons that software engineering learned decades ago: modular code, dependency management, automated testing, documentation, and CI/CD. Whether you use dbt, DLT, Dataform, or raw SQL — the underlying principles are the same. Understanding why these tools exist matters more than mastering any one of them.
