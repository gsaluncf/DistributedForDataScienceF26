dbt Quickstart: PyCharm & Databricks

A fast, command-line focused workflow for configuring dbt locally with PyCharm and connecting it to a Databricks environment using the built-in samples.tpch dataset.

Step 1 Project Setup & Virtual Environment

We are going to build a clean directory and an isolated Python environment entirely from scratch so that there are no missing steps. Open your terminal (or PyCharm's built-in Terminal tab) and run the following exact sequence of commands:

1. Create and enter the project folder

mkdir dbt_lab
cd dbt_lab

2. Create the virtual environment

Windows:

py -3.12 -m venv .venv

Mac / Linux:

python3 -m venv .venv

3. Activate the virtual environment

Windows:

.venv\Scripts\activate

Mac / Linux:

source .venv/bin/activate

Before proceeding, ensure you see (.venv) prepended to your terminal prompt!

4. Install the dbt adapter

This single pip command will install both the Databricks adapter and the core dbt engine as a dependency.

pip install dbt-databricks

Step 2 Connecting to Databricks (dbt init)

Preparation: Gather your Databricks Credentials
  1. In Databricks, click the Compute icon in the sidebar, then navigate to the SQL warehouses tab.
  2. Start your Serverless Starter Warehouse.
  3. Navigate to the Connection details tab.
  4. Click the orange *dbt logo button to pop open the dbt configuration side-panel.
  5. Click Generate new token. Databricks will generate a block of YAML code that looks exactly like this:
default:
  target: dev
  outputs:
    dev:
      type: databricks
      schema: default
      host: dbc-797e799a-af03.cloud.databricks.com
      http_path: /sql/1.0/warehouses/79d7e46984c59aa2
      token: xxxxx

Copy this block of YAML code and save it somewhere handy (like a temporary Notepad file). You will need these three specific values: host, http_path, and token.

Return to your PyCharm Terminal and run the initialization command to set up the dbt folder structure:

dbt init my_project

It will prompt you for several inputs. Read the terminal carefully. Here is exactly how to answer them:

  • Enter a number (database option): 1 (Important: Select databricks, do NOT select spark!)
  • host (yourorg.databricks.com): Paste the host value you saved earlier
  • http_path (HTTP Path): Paste the http_path value you saved earlier
  • Desired access token option (enter a number): 1 (to correctly select "use access token")
  • token (dapiXXX...): Paste the actual token value you saved earlier
  • Desired unity catalog option (enter a number): 1 (to use Unity Catalog)
  • catalog (initial catalog): workspace (or whatever your default catalog is named)
  • schema (default schema): dbt_USERNAME (Use your own name/IDs so you don't overwrite other students!)
  • threads (1 or more): 1

Move your terminal into the new project directory:

cd my_project

Step 3 The 'dbt run' Command

Your initialized project comes with two test models inside the models/example/ directory. Run them right now from your terminal to verify your connection to Databricks is working:

dbt run

If you get Completed successfully, dbt has automatically compiled SQL and built two tables over in Databricks! You can now delete the models/example directory. We are going to build our own models.

Step 4 Adding Sources

By default, if you want your SQL to read data from a data warehouse, you have to hardcode the full path to the table (e.g. FROM samples.tpch.orders). In dbt, we centralize these raw, upstream tables into Sources instead.

Declaring a Source does not create any data. It simply tells dbt that external data already exists, allowing dbt to:

  • Let you reference the table dynamically using the {{ source() }} macro instead of brittle hardcoded paths.
  • Place the raw data onto your automatically generated Data Lineage graphs.
  • Run data quality tests against the raw data before running your pipeline.
  • Automate "freshness" checks to ensure the data is up-to-date.

Databricks comes with a built-in retail dataset stored in the samples.tpch catalog schema. To declare this existing raw data in dbt, create a file named models/sources.yml:

version: 2

sources:
  - name: tpch
    catalog: samples
    schema: tpch
    tables:
      - name: orders
      - name: customer

To verify that dbt has successfully registered these new external sources, you can ask dbt to list all known sources in your project by running the following command in your terminal:

dbt ls --resource-type source

You should see something exactly like this print to your terminal, proving dbt is now fully aware of the Databricks data:

source:ds326project.tpch.customer
source:ds326project.tpch.orders

Step 5 Practical Analytical Queries

1. Staging Model (Silver)

Create a file at models/staging_orders.sql. We use the {{ source() }} macro to reference the raw table defined in our YAML file above.

WITH raw_orders AS (
    SELECT * FROM {{ source('tpch', 'orders') }}
)

SELECT
    o_orderkey AS order_id,
    o_custkey AS customer_id,
    o_orderstatus AS status,
    o_orderdate AS order_date,
    o_totalprice AS total_price
FROM raw_orders

2. Analytical Model (Gold)

Create a file at models/monthly_orders.sql. We use the {{ ref() }} macro to build downstream from our staging table, forcing a dependency.

{{ config(materialized='table') }}

WITH transformed_orders AS (
    SELECT 
        DATE_TRUNC('month', order_date) AS order_month,
        status,
        order_id
    FROM {{ ref('staging_orders') }}
)

SELECT 
    order_month,
    COUNT(order_id) AS total_orders
FROM transformed_orders
GROUP BY order_month
ORDER BY order_month DESC

Execute the pipeline from terminal. dbt resolves the DAG of dependencies, executing the staging view first and the analytical table second:

dbt run

Viewing the Compiled SQL

Curious what dbt actually sent to Databricks? Look inside your PyCharm file tree at a newly created directory similar to ds326project\target\compiled. dbt saves the final "pure SQL" version of every model here.

Step 6 Generate Lineage & Documentation

Because you've parameterized your logic using {{ source() }} and {{ ref() }}, dbt can trace the flow of data end-to-end and natively build a documentation website exploring your schema graph.

dbt docs generate
dbt docs serve

A browser window will open. Click the green "View Lineage Graph" icon in the bottom right corner to visually observe the automated data flow. You will clearly see the original raw source table securely linked into your staging view, which in turn flows right into the final analytical dataset. This interactive DAG of your compiled code is a primary reason Data Engineers love dbt.

Step 7 Jinja Expressions

What if we want monthly totals broken down by the specific status flags ('O' for Open, 'F' for Final, 'P' for Pending)? We can use a Python-style Jinja loop to dynamically compile repetitive SQL logic.

Update your models/monthly_orders.sql file to this version:

{{ config(materialized='table') }}

{% set statuses = ['O', 'F', 'P'] %}

WITH transformed_orders AS (
    SELECT 
        DATE_TRUNC('month', order_date) AS order_month,
        status,
        order_id
    FROM {{ ref('staging_orders') }}
)

SELECT 
    order_month,
    COUNT(order_id) AS total_orders,
    
    {% for status in statuses %}
    COUNT(CASE WHEN status = '{{ status }}' THEN order_id END) AS status_{{ status }}_count
    {% if not loop.last %},{% endif %}
    {% endfor %}

FROM transformed_orders
GROUP BY order_month
ORDER BY order_month DESC

Notice the special {% if not loop.last %},{% endif %} statement inside the loop. When dbt essentially writes out COUNT(CASE...) over and over, we need commas to separate the columns. This special Jinja logic tells dbt: "Add a comma here, unless it's the final iteration of the loop." This prevents a trailing comma syntax error in the compiled SQL.

Run dbt run again to execute this macro.

For a deeper dive into macros, variables, and dicts, check out the official dbt Jinja Documentation.

Appendix Core Commands & Config Explained

What the Commands Do

  • dbt init: Bootstraps a new project. It creates the required folder structure (models, target, macros) and prompts you for credentials to generate a profiles.yml file connecting dbt to your data warehouse.
  • dbt run: The workhorse command. It parses your SQL models, builds a dependency graph (DAG) based on your {{ ref() }} calls, compiles the Jinja loops into pure SQL, and executes the queries against Databricks to construct your tables/views.
  • dbt test (Not used above): Runs data quality tests. If you define tests (like unique or not_null) in a YAML file, dbt compiles them into SELECT queries. If any query returns rows, the test fails.
  • dbt docs generate: Parses your models, descriptions, and tests into a JSON manifest file (saving it in the target/ directory).
  • dbt docs serve: Spins up a local web server to display the JSON manifest generated in the previous step as an interactive HTML website.

Understanding {{ config() }} Options

By default, dbt wraps all your SELECT statements in a CREATE VIEW AS .... When you add the {{ config() }} block at the top of a file, you can override how Databricks physically materializes the data.

  • {{ config(materialized='view') }} (Default): Creates a logical view. The query runs every time a user selects from it. Slower to query downstream, but takes no storage and is always perfectly up to date. Excellent for Staging models.
  • {{ config(materialized='table') }}: Wraps your query in CREATE TABLE AS .... Drops the old table and rebuilds it entirely from scratch. Fast to query downstream, but costs storage and compute to rebuild. Ideal for Analytical (Gold) models.
  • {{ config(materialized='incremental') }}: Advanced mode. Instead of dropping the table, dbt injects an INSERT INTO or MERGE statement, only processing new records that arrived since the last run. Used for massive event-stream tables.
  • {{ config(materialized='ephemeral') }}: Doesn't create an object in the database at all. Instead, it injects the model's SQL as a Common Table Expression (CTE) into whatever downstream models reference it.
Summary

In this quick lab, you connected your local PyCharm environment to Databricks using dbt-databricks. You defined raw sources from the Databricks catalog using sources.yml and built structured staging and analytical models using SQL and ref() macros. Finally, you utilized Jinja templating dynamically generating complex SQL logic and viewed an interactive DAG.
