Week 3 Lecture 2: Persistence with DynamoDB

In Lecture 1, we learned how to build serverless APIs with Lambda and API Gateway. But what happens when your Lambda function finishes executing? All data in memory is lost. To build real applications, we need persistent storage. That's where Amazon DynamoDB comes in.

DynamoDB is AWS's fully managed NoSQL database service. It's designed for applications that need consistent, single-digit millisecond latency at any scale. In this lecture, we'll explore how DynamoDB works, how to design tables effectively, and how to integrate it with Lambda functions.

Part 1: What is DynamoDB?

NoSQL vs SQL Databases

Before diving into DynamoDB, let's understand the difference between SQL and NoSQL databases:

Aspect SQL (Relational) NoSQL (DynamoDB)
Data Model Tables with fixed schemas, rows and columns Key-value and document store, flexible schema
Relationships Foreign keys, JOINs across tables Denormalized data, no JOINs
Scaling Vertical (bigger servers) Horizontal (more partitions)
Query Language SQL (SELECT, JOIN, WHERE) API calls (get_item, put_item, query)
Best For Complex queries, transactions, reporting High-speed reads/writes, simple queries, massive scale

DynamoDB Key Features

Feature Description
Fully Managed No servers to provision, patch, or manage
Performance Single-digit millisecond latency at any scale
Scalability Automatically scales to handle trillions of requests per day
Flexible Schema Each item can have different attributes
Built-in Security Encryption at rest, IAM integration, VPC endpoints

Part 2: Table Design - Keys and Access Patterns

Understanding Partition Keys and Sort Keys

DynamoDB tables are organized around keys. Every table must have a partition key (also called a hash key), and optionally a sort key (also called a range key).

Key Type Purpose Example
Partition Key Determines which partition stores the item. Must be unique if no sort key. user_id, order_id, product_id
Sort Key Sorts items with the same partition key. Enables range queries. timestamp, date, version
Composite Key Partition key + sort key together form a unique identifier user_id + timestamp

Access Patterns: Design for Your Queries

Unlike SQL databases where you design normalized tables first and then write queries, with DynamoDB you design your table based on your access patterns. You need to know how you'll query the data before you create the table.

Common Access Patterns

Property Value Explanation
Pattern Key Design Example
Get single item Partition key only Get user by user_id
Get related items Partition key + sort key Get all orders for user_id
Range queries Partition key + sort key Get orders between dates

Our simple table uses only a partition key (id) because our access pattern is straightforward: retrieve a single item by its ID.

Part 3: Throughput Models - On-Demand vs Provisioned

Understanding Capacity Modes

DynamoDB offers two capacity modes that determine how you pay for read and write throughput:

Aspect On-Demand (PAY_PER_REQUEST) Provisioned
Pricing Model Pay per request (read/write) Pay for provisioned capacity (hourly)
Capacity Planning None required, auto-scales Must specify read/write capacity units
Best For Unpredictable workloads, new applications, development Predictable traffic, cost optimization
Throttling Risk Very low (handles spikes automatically) Higher (if traffic exceeds provisioned capacity)
Cost Higher per request, but no waste Lower per request, but pay even if unused

Why We Use On-Demand Mode

Our SAM template uses PAY_PER_REQUEST (on-demand mode) because:

1. We don't know how many students will deploy this lab
2. Traffic is unpredictable during testing and development
3. No capacity planning required - perfect for learning
4. You only pay for what you use (likely pennies for this lab)

Understanding Hot Partitions

A hot partition occurs when one partition receives significantly more traffic than others. This can cause performance issues and throttling.

Scenario Problem Solution
Celebrity User One user_id gets millions of requests Use caching (ElastiCache), add random suffix to partition key
Sequential IDs All new items go to same partition (e.g., timestamp-based keys) Use UUIDs or add random prefix
Uneven Access Some items accessed far more than others Use DAX (DynamoDB Accelerator) for caching

Best Practice: Distribute Partition Keys Evenly

Choose partition keys that distribute requests evenly across partitions. Good examples: user_id (UUID), order_id (UUID), email address. Bad examples: status (only a few values), date (all writes go to today's partition).

DynamoDB Operations in Python (Boto3)

Our Lambda handler uses the Boto3 library to interact with DynamoDB. Let's examine the key operations:

1. Initialization (Outside Handler)

import boto3
import os

# Initialize once per container (cold start only)
dynamodb = boto3.resource('dynamodb')
table = dynamodb.Table(os.environ.get('TABLE_NAME', 'items'))
Code Explanation
boto3.resource('dynamodb') Creates a high-level DynamoDB resource object
os.environ.get('TABLE_NAME') Reads table name from environment variable (set by SAM)
dynamodb.Table(...) Creates a Table resource for performing operations

2. PUT Operation (Create Item)

def handle_post(event):
    body = json.loads(event.get('body', '{}'))
    item_id = body.get('id')

    # Convert floats to Decimal for DynamoDB
    item = convert_floats(body)

    # Store item in DynamoDB
    table.put_item(Item=item)

    return {
        'statusCode': 201,
        'body': json.dumps({'message': 'Item created', 'id': item_id})
    }
Step What Happens
1. Parse JSON Convert request body string to Python dict
2. Convert floats DynamoDB requires Decimal type for numbers (not float)
3. put_item() Writes item to DynamoDB (creates or replaces)
4. Return 201 HTTP 201 Created status code

3. GET Operation (Retrieve Item)

def handle_get(event):
    item_id = event.get('pathParameters', {}).get('id')

    # Get item from DynamoDB
    response = table.get_item(Key={'id': item_id})

    if 'Item' in response:
        # Convert Decimal to float for JSON serialization
        item = convert_decimals(response['Item'])
        return {
            'statusCode': 200,
            'body': json.dumps(item)
        }
    else:
        return {
            'statusCode': 404,
            'body': json.dumps({'error': 'Item not found'})
        }
Step What Happens
1. Extract ID Get item ID from URL path parameters
2. get_item() Retrieve item by partition key
3. Check if found Response contains 'Item' key if found
4. Convert Decimals JSON doesn't support Decimal, convert to float
5. Return result 200 if found, 404 if not found

The Decimal Problem and Solution

One of the most common issues when working with DynamoDB in Python is the Decimal type mismatch. Here's why it happens and how we solve it:

The Problem

JSON uses: float for numbers (e.g., 29.99)
DynamoDB requires: Decimal for numbers (to avoid floating-point precision errors)
Result: If you try to store a float in DynamoDB, you get a TypeError

The Solution: Helper Functions

We use two helper functions to convert between float and Decimal:

from decimal import Decimal

def convert_floats(obj):
    """Recursively convert float to Decimal for DynamoDB storage"""
    if isinstance(obj, list):
        return [convert_floats(i) for i in obj]
    elif isinstance(obj, dict):
        return {k: convert_floats(v) for k, v in obj.items()}
    elif isinstance(obj, float):
        return Decimal(str(obj))  # Convert float to Decimal
    else:
        return obj

def convert_decimals(obj):
    """Recursively convert Decimal to float for JSON serialization"""
    if isinstance(obj, list):
        return [convert_decimals(i) for i in obj]
    elif isinstance(obj, dict):
        return {k: convert_decimals(v) for k, v in obj.items()}
    elif isinstance(obj, Decimal):
        return float(obj)  # Convert Decimal to float
    else:
        return obj
Function When to Use Why
convert_floats() Before writing to DynamoDB DynamoDB requires Decimal for numbers
convert_decimals() After reading from DynamoDB JSON doesn't support Decimal type

Why Recursive?

The functions are recursive because your data might have nested structures. For example:

{
  "id": "product-123",
  "name": "Widget",
  "price": 29.99,
  "dimensions": {
    "width": 10.5,
    "height": 5.25
  },
  "prices": [19.99, 29.99, 39.99]
}

All the floats (29.99, 10.5, 5.25, 19.99, etc.) need to be converted, no matter how deeply nested they are.

Part 5: Best Practices and Common Patterns

DynamoDB Best Practices

Best Practice Why It Matters
Design for access patterns first DynamoDB is optimized for specific query patterns, not ad-hoc queries
Use UUIDs for partition keys Ensures even distribution across partitions
Initialize boto3 outside handler Reuse connections across warm starts for better performance
Handle Decimal conversion Avoid TypeError when working with numbers
Use on-demand for unpredictable workloads Avoid throttling and capacity planning complexity
Add error handling DynamoDB operations can fail (network issues, throttling, etc.)

Common DynamoDB Operations

Operation Boto3 Method Performance Use Case
Create/Update put_item() Fast (single-digit ms) Write a single item (creates or replaces)
Read get_item() Fast (single-digit ms) Retrieve a single item by key
Query query() Fast (reads one partition) Get multiple items with same partition key
Scan scan() SLOW (reads entire table) Read entire table (expensive, avoid in production)
Delete delete_item() Fast (single-digit ms) Remove a single item by key
Batch Write batch_write_item() Efficient (up to 25 items) Write up to 25 items in one request

Warning: Avoid scan() in Production

scan() reads EVERY item in the table, which is slow and expensive. For a table with 1 million items, scan() reads all 1 million items even if you only need 10 results. Use query() with proper partition keys instead.

Example: If scan() takes 10 seconds on a 100,000-item table, it will take 100 seconds on a 1,000,000-item table. query() stays fast regardless of table size.

DynamoDB Pricing and Cost Considerations

Understanding DynamoDB pricing helps you build cost-effective applications:

Cost Component On-Demand Pricing How to Minimize
Read Requests $0.25 per million requests Use get_item() instead of scan(), cache frequently accessed data
Write Requests $1.25 per million requests Use batch_write_item() for bulk operations
Storage $0.25 per GB-month Delete old data, use TTL for automatic expiration
Data Transfer Free within same region Keep Lambda and DynamoDB in same region

Cost Example for This Lab

If you make 100 requests during the lab (50 writes, 50 reads), your cost will be approximately $0.00008 (less than 1 cent). DynamoDB's free tier includes 25 GB storage and 25 million read/write requests per month, so this lab will likely cost you nothing.

When NOT to Use DynamoDB

DynamoDB is powerful, but it's not the right choice for every use case:

Use Case Why DynamoDB is NOT Ideal Better Alternative
Complex JOINs No JOIN support, requires denormalization PostgreSQL (RDS), MySQL (RDS)
Ad-hoc Queries Must design for specific access patterns SQL databases with flexible querying
Analytics/Reporting Not optimized for aggregations, GROUP BY Amazon Redshift, Athena
Full-Text Search No built-in search capabilities Amazon OpenSearch, Elasticsearch
Graph Relationships Not designed for traversing relationships Amazon Neptune (graph database)

DynamoDB excels at: High-speed key-value lookups, user sessions, shopping carts, gaming leaderboards, IoT data, and mobile app backends where you know your access patterns in advance.
