Lecture 1: Serverless Architecture & AWS SAM

This week marks a major shift in how we build applications. Last week, you deployed a static website to S3. This week, you'll build a complete serverless API that can handle user requests, execute business logic, and persist data to a database. Instead of managing servers, you'll write code that AWS runs automatically in response to HTTP requests.

We'll use AWS SAM (Serverless Application Model) to deploy everything with just 3 commands instead of 25+ manual steps.

Part 1: From Client-Server to Serverless

Week 2 Recap: Static Website Architecture

Last week, we built a static website hosted on S3. This architecture had clear limitations:

Component What It Does Limitation
Client (Browser) Requests HTML/CSS/JS files All logic runs in browser
Server (S3) Serves static files No backend processing
Database None No persistent storage

This works for portfolios and documentation sites, but real applications need backend capabilities like user authentication, database queries, business logic, and API integrations.

Traditional Server Architecture vs. Serverless

In a traditional setup, you run a web server (Node.js, Django, Flask) on a virtual machine. You're responsible for everything.

Concern Traditional Servers Serverless
Provisioning Choose instance types, configure OS, install dependencies AWS handles automatically
Scaling Manually add/remove servers based on traffic Automatic (1 to 10,000 requests)
Maintenance Patch OS, update libraries, monitor health AWS handles automatically
Cost Pay 24/7, even when idle Pay only for execution time (milliseconds)
Availability Handle crashes, load balancing, failover Built-in redundancy across data centers

Important: "Serverless" Doesn't Mean No Servers

Servers still exist. AWS runs your code on physical machines. The term "serverless" means you don't manage servers. AWS handles provisioning, scaling, patching, and monitoring automatically.

Part 2: The AWS Ecosystem - A Lot of Products

The Challenge: AWS Has 200+ Services

AWS offers an overwhelming number of services. For our serverless API, we need just 3 core services:

Service Purpose Analogy
AWS Lambda Runs your code (compute) The engine that processes requests
API Gateway Handles HTTP requests (routing) The front door that receives requests
DynamoDB Stores data (database) The filing cabinet for persistent storage

But to make these work together, you also need supporting services like IAM (permissions), CloudWatch (logs), and CloudFormation (infrastructure as code).

The Manual Way: 25+ Steps

If you were to set this up manually using the AWS CLI or Console, you'd need to execute over 25 commands in the correct order. Here's a sample:

Step Command What It Does
1 aws dynamodb create-table... Create DynamoDB table
2-5 aws iam create-role... Create IAM role and policies
6-8 zip, aws lambda create-function... Package and deploy Lambda
9-15 aws apigatewayv2 create-api... Create API Gateway and routes
16+ aws lambda add-permission... Grant permissions, configure CORS, test

This is tedious, error-prone, and hard to reproduce. If you make a mistake, you have to manually clean up and start over.

Infrastructure as Code: A Better Approach

Infrastructure as Code (IaC) means defining your infrastructure in a configuration file instead of running manual commands. Benefits include:

Benefit Description
Reproducible Deploy the same infrastructure multiple times (dev, staging, prod)
Version Controlled Track changes in Git, review with pull requests
Automated Deploy with CI/CD pipelines
Self-Documenting The template shows exactly what's deployed

Part 3: AWS SAM - Serverless Made Easy

What is AWS SAM?

AWS SAM (Serverless Application Model) is an open-source framework for building serverless applications. It's an extension of AWS CloudFormation with simplified syntax for serverless resources.

Feature CloudFormation AWS SAM
Syntax Verbose, requires many lines Simplified for serverless
Local Testing Not supported sam local start-api
Deployment Manual packaging required sam build & sam deploy
Best For Any AWS infrastructure Serverless applications

Installing AWS SAM CLI

Required: Install SAM CLI

Before you can deploy serverless applications, you need to install the AWS SAM CLI. Follow the official installation guide:

AWS SAM CLI Installation Guide

After installation, verify with: sam --version

The SAM Workflow: 3 Simple Commands

Instead of 25+ manual steps, SAM reduces deployment to just 3 commands:

Command What It Does
sam build Packages your Lambda code and dependencies
sam deploy --guided Deploys your application to AWS (prompts for configuration)
sam delete Removes all deployed resources (cleanup)

That's it! SAM handles all the complexity: creating IAM roles, configuring API Gateway, deploying Lambda, setting up DynamoDB, and wiring everything together.

Part 4: Lambda Functions - How They Work

What is AWS Lambda?

AWS Lambda is a compute service that runs your code in response to events. You upload your code, and Lambda handles everything else: provisioning servers, scaling, patching, monitoring.

AWS Lambda execution model (Source: AWS)

The Lambda Handler Pattern

Every Lambda function has a handler - the entry point that AWS calls when your function is invoked:

def lambda_handler(event, context):
    """
    event: Contains data about the triggering event (HTTP request, S3 upload, etc.)
    context: Provides runtime information (request ID, memory limit, etc.)
    """
    return {
        'statusCode': 200,
        'body': 'Hello from Lambda!'
    }
Parameter Description Example
event Data about the triggering event event['body'], event['pathParameters']
context Runtime information context.request_id, context.memory_limit_in_mb

Lambda Execution Model

Understanding how Lambda executes your code is crucial for writing efficient functions:

Phase What Happens Duration
Cold Start Download code, start runtime, initialize global variables 100-1000ms
Warm Start Reuse existing container, skip initialization 1-10ms
Execution Run your handler function Depends on your code

Optimization Tip: Initialize Outside the Handler

Code outside the handler runs once per container (cold start). Code inside the handler runs on every invocation. Initialize database connections, import libraries, and load configuration outside the handler for better performance.

# GOOD: Initialize once (cold start only)
import boto3
dynamodb = boto3.resource('dynamodb')
table = dynamodb.Table('items')

def lambda_handler(event, context):
    # This runs on every invocation
    table.put_item(Item={'id': '123'})
    return {'statusCode': 200}

# BAD: Initialize on every invocation
def lambda_handler(event, context):
    import boto3  # Slow!
    dynamodb = boto3.resource('dynamodb')  # Slow!
    table = dynamodb.Table('items')  # Slow!
    table.put_item(Item={'id': '123'})
    return {'statusCode': 200}

Part 5: REST APIs and HTTP Basics

API Gateway: HTTP API vs REST API

AWS API Gateway offers two types of APIs. Our project uses HTTP API because it's simpler and cheaper:

Feature HTTP API REST API
Cost $1.00 per million requests $3.50 per million requests
Latency Lower latency Slightly higher
Features Basic routing, CORS, JWT authorization Advanced features: API keys, usage plans, request validation
Best For Most serverless applications Complex enterprise APIs with throttling/quotas

Why We Use HTTP API

For this course, HTTP API is perfect: it's 70% cheaper, faster, and has all the features we need. Unless you specifically need advanced features like API keys or usage plans, HTTP API is the better choice.

What is a REST API?

REST (Representational State Transfer) is an architectural style for building APIs. It uses HTTP methods to perform operations on resources.

HTTP Method Purpose Example
GET Retrieve data GET /items/123
POST Create new data POST /items
PUT Update existing data PUT /items/123
DELETE Remove data DELETE /items/123

HTTP Request/Response Cycle

When a client makes an HTTP request, here's what happens:

Step Component Action
1 Client Sends HTTP request (method, path, headers, body)
2 API Gateway Receives request, matches route, creates event object
3 Lambda Processes event, executes business logic
4 API Gateway Converts Lambda response to HTTP response
5 Client Receives HTTP response (status code, headers, body)

Part 6: Building a Complete Lambda Handler

Our Project Structure

For this week's lab, you'll deploy a serverless API with two endpoints:

Endpoint Method Purpose
POST /items POST Create a new item in DynamoDB
GET /items/{id} GET Retrieve an item from DynamoDB by ID

The Lambda Handler Code

Here's the complete handler with routing, error handling, and DynamoDB integration:

import json
import os
import boto3
from decimal import Decimal

# Initialize outside handler (runs once per container)
dynamodb = boto3.resource('dynamodb')
table = dynamodb.Table(os.environ.get('TABLE_NAME', 'items'))

def lambda_handler(event, context):
    """Main handler - routes requests based on HTTP method"""
    http_method = event.get('requestContext', {}).get('http', {}).get('method')

    if http_method == 'GET':
        return handle_get(event)
    elif http_method == 'POST':
        return handle_post(event)
    else:
        return {'statusCode': 405, 'body': json.dumps({'error': 'Method not allowed'})}

def handle_get(event):
    """Retrieve an item by ID"""
    item_id = event.get('pathParameters', {}).get('id')

    if not item_id:
        return {'statusCode': 400, 'body': json.dumps({'error': 'Missing id parameter'})}

    try:
        response = table.get_item(Key={'id': item_id})

        if 'Item' in response:
            item = convert_decimals(response['Item'])
            return {'statusCode': 200, 'body': json.dumps(item)}
        else:
            return {'statusCode': 404, 'body': json.dumps({'error': 'Item not found'})}
    except Exception as e:
        return {'statusCode': 500, 'body': json.dumps({'error': str(e)})}

def handle_post(event):
    """Create a new item"""
    try:
        body = json.loads(event.get('body', '{}'))
        item_id = body.get('id')

        if not item_id:
            return {'statusCode': 400, 'body': json.dumps({'error': 'Missing id in request body'})}

        # Convert floats to Decimal for DynamoDB
        item = convert_floats(body)
        table.put_item(Item=item)

        return {'statusCode': 201, 'body': json.dumps({'message': 'Item created', 'id': item_id})}
    except Exception as e:
        return {'statusCode': 500, 'body': json.dumps({'error': str(e)})}

def convert_floats(obj):
    """Recursively convert float to Decimal for DynamoDB"""
    if isinstance(obj, list):
        return [convert_floats(i) for i in obj]
    elif isinstance(obj, dict):
        return {k: convert_floats(v) for k, v in obj.items()}
    elif isinstance(obj, float):
        return Decimal(str(obj))
    else:
        return obj

def convert_decimals(obj):
    """Recursively convert Decimal to float for JSON serialization"""
    if isinstance(obj, list):
        return [convert_decimals(i) for i in obj]
    elif isinstance(obj, dict):
        return {k: convert_decimals(v) for k, v in obj.items()}
    elif isinstance(obj, Decimal):
        return float(obj)
    else:
        return obj

Key Concepts in the Handler

Concept Why It Matters
Initialization Outside Handler boto3 and DynamoDB table are initialized once per container, not on every request
Environment Variables TABLE_NAME is injected by SAM, making code reusable across environments
Routing Logic Single handler routes to different functions based on HTTP method
Error Handling Try/except blocks catch errors and return proper HTTP status codes
Decimal Conversion DynamoDB requires Decimal for numbers; JSON requires float
JSON Parsing event['body'] is a string that must be parsed with json.loads()

The SAM Template

Here's the complete SAM template that defines all infrastructure:

AWSTemplateFormatVersion: '2010-09-09'
Transform: AWS::Serverless-2016-10-31
Description: Week 3 - Serverless API

Globals:
  Function:
    Timeout: 3
    MemorySize: 128
    Runtime: python3.12

Resources:
  # DynamoDB Table
  ItemsTable:
    Type: AWS::DynamoDB::Table
    Properties:
      TableName: !Sub '${AWS::StackName}-items'
      AttributeDefinitions:
        - AttributeName: id
          AttributeType: S
      KeySchema:
        - AttributeName: id
          KeyType: HASH
      BillingMode: PAY_PER_REQUEST

  # Lambda Function
  ApiFunction:
    Type: AWS::Serverless::Function
    Properties:
      CodeUri: src/
      Handler: handler.lambda_handler
      Environment:
        Variables:
          TABLE_NAME: !Ref ItemsTable
      Policies:
        - DynamoDBCrudPolicy:
            TableName: !Ref ItemsTable
      Events:
        CreateItem:
          Type: HttpApi
          Properties:
            Path: /items
            Method: POST
        GetItem:
          Type: HttpApi
          Properties:
            Path: /items/{id}
            Method: GET

Outputs:
  ApiUrl:
    Value: !Sub 'https://${ServerlessHttpApi}.execute-api.${AWS::Region}.amazonaws.com'
  FunctionName:
    Value: !Ref ApiFunction
  TableName:
    Value: !Ref ItemsTable

Notice how SAM automatically creates the API Gateway, wires it to Lambda, sets up IAM permissions, and passes the table name as an environment variable. All of this would require 25+ manual commands without SAM!

Part 7: Security and IAM

How SAM Handles Security

One of SAM's biggest benefits is automatic IAM role creation. Let's understand what's happening behind the scenes:

SAM Template Line What SAM Creates
DynamoDBCrudPolicy IAM role with permissions for dynamodb:PutItem, GetItem, UpdateItem, DeleteItem, Query, Scan
TableName: !Ref ItemsTable Restricts permissions to ONLY this specific table (least privilege)
Events: HttpApi Lambda resource policy allowing API Gateway to invoke the function

Principle of Least Privilege

SAM follows security best practices by granting the Lambda function ONLY the permissions it needs. The function can access the DynamoDB table, but cannot create new tables, access S3, or perform other AWS operations.

Real-World Security Considerations

Security Layer What You'd Add in Production
Authentication JWT tokens, AWS Cognito, or API keys to verify user identity
Authorization Check if authenticated user has permission to access specific resources
Input Validation Validate request data to prevent injection attacks
Rate Limiting Throttle requests to prevent abuse and control costs
Encryption HTTPS for data in transit (automatic), DynamoDB encryption at rest (enabled by default)

For this lab: We're building a learning project, so we skip authentication to keep things simple. In production, you'd never deploy a public API without authentication!
