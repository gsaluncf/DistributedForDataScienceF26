Introduction: Your First Distributed System

Welcome to your first hands-on experience with distributed computing!

In this walkthrough, you'll set up GitHub Pages - a free web hosting service that demonstrates core distributed systems concepts:

  • Replication - Your code exists on your local machine AND on GitHub's servers
  • Synchronization - Push and pull keep both copies in sync
  • Remote Execution - GitHub's servers build and serve your website automatically
  • Global Distribution - Your site is served from GitHub's CDN (Content Delivery Network) worldwide

Why GitHub Pages? It's the perfect "low-hanging fruit" for learning distributed computing - you get immediate, visible results (a live website!) while experiencing fundamental distributed systems concepts.

What You'll Accomplish

By the end of this walkthrough, you will:

  1. Create a public GitHub repository for class projects
  2. Enable GitHub Pages to host a live website

Challenge: Experience the Distributed Workflow

Your Mission

Now that your repository exists on GitHub's servers, let's experience the distributed workflow:

  1. Clone - Copy the remote repository to your local machine
  2. Modify - Make changes locally
  3. Push - Send your changes back to GitHub's servers
  4. Verify - Watch GitHub Pages automatically update your live site

This is the essence of distributed computing! Multiple copies of data, synchronized across different machines.

Commands You'll Use

Open your terminal (Command Prompt, PowerShell, or Git Bash on Windows; Terminal on Mac/Linux) and follow these steps:

Step 1: Clone the Repository to Your Local Machine

cd ~
# Navigate to your home directory (or wherever you want to store your projects)

git clone https://github.com/[your-username]/distributed-computing-projects.git
# Replace [your-username] with your actual GitHub username
# This downloads a complete copy of the repository to your machine

cd distributed-computing-projects
# Enter the newly created directory

What Just Happened?

git clone created a complete copy of your repository on your local machine, including:

  • All files (README.md)
  • All commit history
  • All branches
  • A connection to the remote repository - Git automatically configured a "remote" called origin pointing to GitHub

Distributed replication in action! You now have two copies: one on GitHub's servers, one on your machine.

Verify the remote connection:

git remote -v
# Shows: origin  https://github.com/[your-username]/distributed-computing-projects.git (fetch)
#        origin  https://github.com/[your-username]/distributed-computing-projects.git (push)

The origin remote was automatically created by git clone - you don't need to set it up manually!

Step 2: Make a Change to README.md

Open README.md in your favorite text editor and add a new section at the bottom:

## Recent Updates

- **[Today's Date]**: Successfully completed the distributed workflow challenge!
  - Cloned repository to local machine

  - Experienced distributed computing firsthand!

Save the file.

Step 3: Stage, Commit, and Push Your Changes

git status
# Check what files have changed (you should see README.md as modified)

git add README.md
# Stage the file for commit

git commit -m "Add Recent Updates section - completed distributed workflow challenge"
# Create a commit with your changes

git push origin main
# Push your commit to GitHub's servers

Understanding "git push origin main"

Let's break down this command word by word:

git push - The command to send commits to a remote repository

origin - The name of the remote repository

  • When you ran git clone, Git automatically created a remote called "origin"
  • "origin" is just a nickname (alias) for the full GitHub URL: https://github.com/[your-username]/distributed-computing-projects.git
  • You could have multiple remotes with different names, but "origin" is the standard default name for the main remote
  • Think of it as: "origin" = "the place this repository originally came from"

main - The name of the branch you're pushing

  • "main" is the default branch name in modern Git repositories
  • This tells Git: "push the commits from my local main branch to the main branch on the origin remote"
  • If you were working on a different branch (like "feature1"), you'd push that branch name instead

In plain English:

"Git, please push my commits from my local main branch to the main branch on the origin remote (GitHub)"

What Happens When You Push?

When you run git push origin main:

  1. Your local Git packages up your new commit(s)
  2. Network transfer - The commit data is sent over the internet to GitHub's servers
  3. GitHub receives the commit and stores it in their Git database
  4. GitHub updates the main branch pointer on their server to point to your new commit
  5. GitHub Pages detects the change and automatically triggers a rebuild
  6. Within 1-2 minutes, your live website updates with the new content

This is distributed synchronization in action! Changes flow from your machine to remote servers, triggering automated processes across GitHub's distributed infrastructure.

Step 4: Verify the Changes

  1. Go to your repository on GitHub: https://github.com/[your-username]/distributed-computing-projects
  2. You should see your new commit and the updated README.md
  3. Wait 1-2 minutes, then visit your GitHub Pages site: https://[your-username].github.io/distributed-computing-projects/
  4. Your live website should now show the "Recent Updates" section!

Congratulations!

You've just experienced the complete distributed computing workflow:

  • Replication - Cloned a remote repository locally
  • Local Processing - Made changes on your machine
  • Synchronization - Pushed changes to remote servers
  • Automated Deployment - GitHub Pages rebuilt and deployed your site automatically
  • Global Distribution - Your changes are now live worldwide via CDN

This is the foundation of modern distributed systems! The same concepts apply to cloud computing, microservices, distributed databases, and more.

Key Commands Explained

  • git clone [url] - Copy a remote repository to your local machine (also automatically sets up "origin" remote)
  • git status - Show which files have changed
  • git add [file] - Stage a file for commit
  • git commit -m "[message]" - Create a commit with your staged changes
  • git push origin main - Send your commits to the remote repository
    • origin = the remote repository (GitHub) - automatically configured by git clone
    • main = the branch name you're pushing
  • git remote -v - Show configured remotes (verify "origin" points to GitHub)

Troubleshooting

Problem: "Permission denied" when pushing

Solution: You need to authenticate with GitHub. Use one of these methods:

  • HTTPS with Personal Access Token - Generate a token at GitHub Settings → Developer settings → Personal access tokens
  • SSH Keys - Set up SSH keys following GitHub's SSH guide

Problem: GitHub Pages site not updating

Solution: Wait 2-3 minutes and refresh. GitHub Pages can take a moment to rebuild. Check Settings → Pages for deployment status.

Problem: "fatal: not a git repository"

Solution: Make sure you're inside the cloned directory. Run cd distributed-computing-projects first.

Problem: Merge conflicts

Solution: If you edited the file both on GitHub and locally, you might get conflicts. Run git pull origin main before making local changes to sync first.
