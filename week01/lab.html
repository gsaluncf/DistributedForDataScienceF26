Week 1 Lab

You'll practice resolving a merge conflict and publishing a simple GitHub Pages site from a branch. Submit your Pull Request link and Pages URL to Canvas.

Setup links (Windows users: PowerShell)

  • Install & configure AWS CLI v2 (includes PowerShell commands).
  • GitHub Classroom — accept the invite link in Canvas.

Key terms

  • Branch: A movable pointer to a commit; lets you work on changes in isolation.
  • Pull Request (PR): A proposal to merge changes into a target branch with review.
  • Merge conflict: When two changes touch the same lines and Git cannot auto-merge.
  • GitHub Pages: Static site hosting directly from a repo branch (e.g., /docs on main).

Part A — Git Conflict Kata (PR with conflict)

  1. Accept the GitHub Classroom assignment (link in Canvas). Clone the repo.
  2. Create a feature branch: git checkout -b feature/update-bio.
  3. Edit the indicated lines in /bio/README.md and commit.
  4. Do not push yet. Switch to main, pull latest, and apply the instructor's conflicting change from upstream (already pushed).
  5. Push your feature branch, open a Pull Request to main. Observe the merge conflict.
  6. Resolve the conflict locally:
    • Checkout your branch; run git fetch origin && git merge origin/main.
    • Open the conflicted file, remove conflict markers, keep intended content.
    • Commit the resolution and push.
  7. Ensure CI (autograding) passes. Request a review; merge the PR.

Check your understanding

  • What causes a merge conflict and how can you minimize them?
  • What is the difference between a fast-forward merge and a merge commit?

Part B — Publish a GitHub Pages site from a branch

  1. Create a folder named docs/ in your repo; add an index.html with your name and course.
  2. Commit and push to main (or use a gh-pages branch).
  3. In GitHub repo → Settings → Pages: set Source to Deploy from a branch, pick main and /docs (or gh-pages root). Save.
  4. Wait for build; visit the Pages URL shown (e.g., https://org.github.io/repo/).
