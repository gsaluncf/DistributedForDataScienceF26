Introduction: Git is a Database

Most people think of Git as a "version control system" - and it is! But understanding what Git actually is will make you a much more confident developer.

Git is a database of snapshots. Every commit is a complete picture of your project at that moment in time. Branches are just pointers (like sticky notes) to commits. The history forms a DAG (Directed Acyclic Graph) where commits point backwards to their parents.

In this walkthrough, you'll:

  • See the raw .git files that Git uses to store everything
  • Prove that branches are just pointers (text files with commit hashes)
  • Explore the Git object database (commits → trees → blobs)
  • Understand the DAG structure and merge commits
  • Learn about reflog - your safety net

The Complete Git Workflow (Local + Remote)

In a typical Git workflow with GitHub (or other remote repositories), you work with both local and remote repositories:

Local Workflow (What We're Focusing On Today)

  1. Working Directory - Your actual files on disk
  2. Staging Area (Index) - Files ready to be committed (git add)
  3. Local Repository - Committed snapshots in .git/ (git commit)

Remote Workflow (For Reference - Not Covered Today)

  1. Remote Repository - Shared repository on GitHub/GitLab/etc.

Common remote commands:

  • git push - Upload your local commits to the remote repository
  • git pull - Download commits from remote and merge them into your current branch
  • git fetch - Download commits from remote but don't merge (safer than pull)
  • git clone - Copy a remote repository to your local machine

Note: This walkthrough focuses on local Git operations only. We're exploring what happens inside the .git directory on your machine. Remote operations (push/pull) build on top of these fundamentals.

Key Concepts: Understanding Git's Vocabulary

Commit

A snapshot - a complete photograph of your entire project at one moment in time. Not the changes you made, but the entire state of every file.

Each commit contains:

  • A pointer to a complete snapshot - every file exactly as it existed
  • Metadata - who created it, when, and the commit message
  • A pointer to the parent commit - the commit that came directly before

Commits point backwards, always backwards. Children know their parents, but parents never know their future children.

Branch

A pointer to a commit. That's it! A branch is literally just a text file containing a 40-character commit hash.

Think of branches as sticky notes you put on commits. When you make a new commit on a branch, Git moves that sticky note forward to point to the new commit.

Example: The file .git/refs/heads/main contains something like: 3b23f2b570aad614eb577f1beeac0e8719851aa6

HEAD

A pointer to a pointer. HEAD tells Git "where you are right now" - which branch you're currently on.

Usually, HEAD points to a branch (like main), and that branch points to a commit. This is two levels of indirection:

HEAD → refs/heads/main → 3b23f2b... (commit hash)

Example: The file .git/HEAD contains: ref: refs/heads/main

Sometimes HEAD can point directly to a commit (called "detached HEAD state"), but usually it points to a branch.

Staging Area (Index)

The "waiting room" for changes before they become a commit. When you run git add, you're putting files in this waiting room.

The staging area is stored in .git/index (a binary file). You can see what's staged with git ls-files --stage.

Three states of a file:

  • Untracked - File exists but Git doesn't know about it
  • Staged - File is in the waiting room (index), ready to be committed
  • Committed - File is safely stored in the Git database

DAG (Directed Acyclic Graph)

The structure of Git history. It's a graph where:

  • Directed - Arrows point in one direction (commits point to parents)
  • Acyclic - No loops (you can't go in circles)
  • Graph - Commits can have multiple children (branches) and multiple parents (merges)

When you merge two branches, you create a merge commit with two parents. This is what makes the graph branch and merge.

Git Objects (The Database)

Git stores everything as objects in .git/objects/. There are three types:

  • Blob - A file's content (just the raw data, no filename)
  • Tree - A directory listing (maps filenames to blobs or other trees)
  • Commit - Points to a tree + metadata + parent commit(s)

The chain: Commit → Tree → Blob

Objects are stored at: .git/objects/<first-2-chars>/<remaining-38-chars>

Files are compressed with zlib. Use git cat-file -p <hash> to read them.

Reflog (Your Safety Net)

A log of everywhere HEAD has been. Every time you commit, checkout, merge, rebase, or do anything that moves HEAD, Git records it in the reflog.

The reflog is stored in .git/logs/HEAD (a plain text file!).

Why it matters: If you accidentally delete a branch or reset to the wrong commit, the reflog lets you find and recover "lost" commits. Nothing is truly lost if it's in the reflog.

Reflog entries expire after 90 days by default, but that's usually plenty of time to recover from mistakes.

Merge Conflict

What happens when Git can't automatically combine changes from two branches. This occurs when:

  • Both branches modified the same lines in the same file
  • Git doesn't know which version to keep

Conflict markers:

<<<<<<< HEAD
Your changes on the current branch
=======
Changes from the branch you're merging
>>>>>>> branch-name

You must manually edit the file to resolve the conflict, then git add and git commit to complete the merge.

The .git Directory

The .git directory is where Git stores EVERYTHING. This is the database!

  • HEAD - Points to current branch
  • objects/ - The database (commits, trees, blobs)
  • refs/ - Branch and tag pointers
  • config - Repository configuration
  • hooks/ - Scripts that run on Git events

Right now it points to refs/heads/main, but that branch doesn't exist yet!

Step 3: Merge with Conflict

What We're Doing

We'll switch back to main, make a conflicting change on the same line, and then merge feature1 into main. This will create a merge commit with two parents, demonstrating the DAG structure.

Commands to Run

git switch main
# Switch back to main branch

# Edit notes.txt to create conflict on line 3
echo "Line 1: hello git" > notes.txt
echo "Line 2: initial snapshot" >> notes.txt
echo "Line 3: change on main - CONFLICT" >> notes.txt

git add notes.txt
# Stage the conflicting change

git commit -m "main change"
# Commit on main branch

git merge feature1
# Try to merge feature1 into main - this will create a CONFLICT

What to Observe

1. Conflict markers in the file:

type notes.txt
Line 1: hello git
Line 2: initial snapshot
<<<<<<< HEAD
Line 3: change on main - CONFLICT
=======
Line 3: change on feature1
>>>>>>> feature1

Understanding Conflict Markers

IMPORTANT: These conflict markers are ACTUALLY IN THE FILE! Git modified notes.txt and inserted these markers directly into the file content.

What each marker means:

  • <<<<<<< HEAD - Start of your current branch's version (main)
  • ======= - Separator between the two conflicting versions
  • >>>>>>> feature1 - End of the incoming branch's version

How to Resolve the Conflict

Step 1: Decide what you want to keep

In this case, we want to keep BOTH lines (both changes are important).

Step 2: Manually edit notes.txt to remove the conflict markers and keep what you want

Open notes.txt in a text editor and change it from this:

Line 1: hello git
Line 2: initial snapshot
<<<<<<< HEAD
Line 3: change on main - CONFLICT
=======
Line 3: change on feature1
>>>>>>> feature1

To this (markers removed, both lines kept):

Line 1: hello git
Line 2: initial snapshot
Line 3: change on main - CONFLICT
Line 3: change on feature1

Step 3: Stage the resolved file

git add notes.txt
# Tell Git "I've resolved the conflict in this file"

Step 4: Complete the merge with a commit

git commit -m "merge feature1 (resolved conflict)"
# This creates the merge commit with TWO parents

2. After resolving the conflict and committing, check the graph:

git log --oneline --decorate --graph --all
*   0f1de9b (HEAD -> main) merge feature1 (resolved conflict)
|\
| * 9e58175 (feature1) feature change
* | 96c9e98 main change
|/
* 3b23f2b initial commit

The DAG (Directed Acyclic Graph)

Look at the graph! The merge commit 0f1de9b has TWO parents:

  • One parent is 96c9e98 (main change)
  • The other parent is 9e58175 (feature change)

This is the DAG structure - commits point backwards to their parents, forming a graph of history.

3. Check the branch pointers again:

type .git\refs\heads\main
0f1de9ba4b5270ed44b2b1e390e3fa750420eeb1

type .git\refs\heads\feature1
9e58175e49bd2a5b344547688c7c68fedc31176f

Notice: main moved forward (new commit), but feature1 stayed the same!

Step 5: Reflog - Your Safety Net

What We're Doing

The reflog (reference log) records every movement of HEAD. This is your safety net - even if you "lose" a commit, it's probably still in the reflog!

Commands to Run

git reflog
# Show the history of where HEAD has pointed - your safety net

type .git\logs\HEAD
# The raw file that stores reflog - Git is just files!

What to Observe

1. The reflog output:

git reflog
0f1de9b HEAD@{0}: commit (merge): merge feature1 (resolved conflict)
96c9e98 HEAD@{1}: commit: main change
3b23f2b HEAD@{2}: checkout: moving from feature1 to main
9e58175 HEAD@{3}: commit: feature change
3b23f2b HEAD@{4}: checkout: moving from main to feature1
3b23f2b HEAD@{5}: commit (initial): initial commit

2. The raw reflog file:

type .git\logs\HEAD
3b23f2b... 3b23f2b... Student <> 1769015310 -0500	checkout: moving from main to feature1
3b23f2b... 9e58175... Student <> 1769015311 -0500	commit: feature change
9e58175... 3b23f2b... Student <> 1769015322 -0500	checkout: moving from feature1 to main
3b23f2b... 96c9e98... Student <> 1769015322 -0500	commit: main change
96c9e98... 0f1de9b... Student <> 1769015339 -0500	commit (merge): merge feature1

Git Rarely Deletes Anything

The reflog shows every movement of HEAD - every commit, every checkout, every merge.

If you accidentally delete a branch or reset to the wrong commit, you can use the reflog to find the "lost" commit hash and recover it!

Example recovery:

git reflog  # Find the commit hash you want
git checkout   # Go back to that commit
git branch recovery-branch  # Create a branch to save it

Advanced Topics: How Git Moves Pointers

Now that you understand Git's internals, let's explore how different Git commands move pointers around. This is the key to understanding merge, rebase, reset, and other advanced operations.

How Merge Moves Pointers

When you merge two branches, Git creates a new commit with TWO parents and moves the current branch pointer to that new commit.

Example scenario:

  • You're on main (pointing to commit A)
  • You run git merge feature1
  • Git creates a new merge commit M with two parents: A (from main) and B (from feature1)
  • Git moves the main pointer to commit M
  • The feature1 pointer stays where it was (pointing to B)
Before merge:
main → A
feature1 → B

After merge:
main → M (merge commit with parents A and B)
feature1 → B (unchanged)

Fast-Forward Merge

If there's no divergence (main hasn't moved since you created the branch), Git can do a fast-forward merge:

  • No new commit is created
  • Git just moves the main pointer forward to point to the same commit as feature1
  • This is just moving a pointer - no new objects created!

What is Rebase?

Rebase "replays" your commits on top of another branch. But here's the critical part: Git cannot move commits. Commits are immutable - their hash is based on their content, metadata, and parent pointer.

What rebase actually does:

  1. Looks at each commit on your branch (e.g., commits B and C)
  2. Calculates the changes (diff) each commit introduced
  3. Creates brand new commits (B' and C') with those same changes, but with a different parent
  4. Moves your branch pointer to the new commits
  5. The old commits (B and C) become orphaned and will eventually be garbage collected
Before rebase:
main → X → Y
feature1 → A → B → C

After rebase:
main → X → Y
feature1 → X → Y → B' → C' (new commits with same changes but different parents)

Old commits B and C still exist (in reflog) but are orphaned

Why "Rebase Rewrites History"

NEVER rebase commits that others have seen!

Because rebase creates new commits with different hashes, if your colleague has the old commits and you push the new ones:

  • Git sees them as completely different commits (different hashes)
  • Your colleague will have duplicates when they pull
  • Conflicts will explode
  • History becomes a mess

Safe use of rebase: Only rebase local branches that you haven't shared with others. It keeps history linear and clean.

Merge vs Rebase: The Trade-off

Merge:

  • Creates a merge commit with two parents
  • Preserves the true history - shows parallel lines of work
  • Safe to use on shared branches
  • History can get messy with lots of merge commits
  • Choosing truth over cleanliness

Rebase:

  • Creates new commits with different parents
  • Makes history linear and clean (looks like work happened sequentially)
  • Dangerous on shared branches (rewrites history)
  • Perfect for cleaning up local work before sharing
  • Choosing a clean story over the messy truth

Other Pointer-Moving Commands

git reset: Moves the current branch pointer to a different commit

  • git reset --soft HEAD~1 - Move branch pointer back one commit, keep changes staged
  • git reset --mixed HEAD~1 - Move branch pointer back, unstage changes (default)
  • git reset --hard HEAD~1 - Move branch pointer back, discard all changes (dangerous!)

git checkout: Moves HEAD to point to a different branch or commit

  • git checkout feature1 - Move HEAD to point to feature1 branch
  • git checkout - Move HEAD to point directly to a commit (detached HEAD)
  • Doesn't move branch pointers, just changes your "view"

git revert: Creates a NEW commit that undoes changes

  • Doesn't move any pointers backwards
  • Safe for shared history - adds corrective commits instead of rewriting
  • git revert - Create new commit that undoes the changes from

Remember: It's All About Pointers

  • Branches are pointers to commits
  • HEAD is a pointer to a branch (or commit)
  • Checkout moves your view (HEAD)
  • Reset moves branch pointers
  • Merge creates a new commit and moves the current branch pointer
  • Rebase creates new commits and moves the current branch pointer
  • Revert adds new commits without moving pointers backwards

And when everything goes wrong? Use git reflog to find the commit you need and recover your work!
