Distributed Cipher — Class Build

We will implement a horizontally scalable keyspace search. Tasks are partitioned and placed on SQS; Lambda workers consume tasks and write results to DynamoDB; logs/artifacts to S3. Each group contributes capacity from their own AWS account with strict cost controls.

Part A — Shared design

  1. Agree on SQS message schema: { "task_id", "range_start", "range_end", "payload" }.
  2. Agree on DynamoDB result schema: PK=task_id, SK=worker_id, attributes: status, attempts, found, output_uri, ttl.
  3. Set SQS visibility timeout (e.g., 30–60s) and redrive to DLQ after N attempts.

Part B — Environment and safety

  1. Per-account Budgets alert at low threshold (e.g., $5).
  2. Lambda reserved concurrency (e.g., 5–10) to cap cost.
  3. DynamoDB TTL attribute configured for temp results (e.g., 7 days).
  4. S3 lifecycle: transition logs to IA after 30 days (optional).

Part C — Implement worker

  1. Template includes /worker/handler.py with TODOs. Implement idempotent processing keyed by task_id.
  2. On success, write result item and upload artifact to S3 (if applicable).
  3. On failure, raise to trigger retry; avoid double-writes using a conditional put.

Check your understanding

  • How does visibility timeout interact with processing time?
  • Why are conditional writes needed for idempotency?

Part D — Load test and observe

  1. Generate N tasks; record SQS queue depth, Lambda invocations, errors.
  2. Observe DynamoDB read/write capacity (throttling?). Adjust batch size or concurrency if needed.
  3. Collect metrics screenshots for submission.

What to submit

  • Group repository URL (template-based) with your worker implementation.
  • Screenshots: SQS metrics, Lambda invocations/concurrency, DynamoDB table view.
  • Brief reflection (<=300 words): scaling behavior and cost learnings.
