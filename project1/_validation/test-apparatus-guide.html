AdFlow Test Apparatus - User Guide

How to use the browser-based testing tool for your Ad Selection Pipeline

What Is the Test Apparatus?

The Test Apparatus is a single HTML file that runs entirely in your browser. It generates realistic ad opportunity messages, sends them to your deployed SQS input queue, waits for your Lambda function to process them, and then drains the results from your SQS results queue. Along the way, it tracks how many messages were sent and received, calculates throughput, and displays each result in a table.

There is no server to install or run. You open the file, enter your AWS credentials and your student ID, and click a button. Everything happens directly between your browser and AWS.

Prerequisites

Before using the test apparatus, you must have:

  1. Completed your Lambda code (all four functions)
  2. Passed all 8 local unit tests (pytest worker/tests/ -v)
  3. Deployed your stack to AWS using sam build and sam deploy --guided
  4. Your AWS Academy Access Key ID and Secret Access Key ready (from Learner Lab)

Opening the Tool

Double-click test-apparatus/index.html in your project folder. It opens in any modern browser (Chrome, Firefox, Edge). No server or terminal needed.

Step 1: Enter Your AWS Credentials

The green card at the top asks for your AWS credentials. These stay entirely within your browser tab and are sent only to AWS (the same way the AWS CLI and boto3 use them). Nothing is stored, logged, or transmitted anywhere else.

Field What to Enter
Access Key ID Starts with AKIA... - copy from your AWS Learner Lab credentials
Secret Access Key The long secret string from the same credentials page (displayed as dots for privacy)
AWS Region Select us-east-1 (the default, matching your deployment)

Click Test Connection to verify your credentials work. The badge should turn green and say "Connected". If it shows "Failed", double-check that you copied the full key and that your Learner Lab session is still active.

Step 2: Enter Your Student ID

In the blue Configuration card, type your student ID (lowercase, the same one you used with sam deploy). For example, if your ID is gsalu, the tool automatically fills in:

Auto-Generated Field Value
Input Queue URL adflow-gsalu-input
Results Queue URL adflow-gsalu-results
DynamoDB Table Name adflow-gsalu-results

The tool resolves these queue names to full SQS URLs automatically when you run a test. You can leave the Random Seed at 42 unless you want different random data.

Step 3: Choose a Test Profile

Test profiles control how many messages are generated and how fast they are sent to your input queue. Start with a small test and work up to larger ones.

Profile Messages Delay When to Use
Warmup 10 500 ms First test after deploying. Verifies your pipeline works at all.
Steady 100 100 ms Moderate load. Good for verifying correctness across many messages.
Burst 500 0 ms All messages at once. Use for the analysis report (Q1-Q4).
Ramp 500 Variable Starts slow, accelerates. Useful for observing Lambda scaling behavior.
Soak 200 1000 ms Sustained trickle. Good for observing steady-state latency.
Custom Your choice Your choice Set your own message count and delay.

Recommended Workflow

  1. Run Warmup (10 messages) first. Wait until all 10 results appear.
  2. Click Full Cleanup in the yellow Cleanup section.
  3. Run Burst (500 messages) for your analysis report data.
  4. Use the same seed (42) each time so your results are reproducible.

Step 4: Run the Test

Click Run Test. Here is what happens behind the scenes:

  1. The tool generates the specified number of ad opportunity messages locally in your browser (using the random seed for reproducibility).
  2. It sends them to your SQS input queue in batches of 10 (respecting the SQS batch limit).
  3. Simultaneously, it begins polling your SQS results queue every 2 seconds.
  4. As results arrive, they appear in the Results table and the Live Metrics update.
  5. The progress bar fills, and when Received equals Sent, the test is complete.
Button What It Does
Run Test Generates messages, sends them to SQS, and starts polling for results.
Stop Stops sending and polling. Messages already in the queue will still be processed by Lambda.
Poll Results Only Does not send any messages. Only polls the results queue. Useful if you stopped a test early and want to pick up remaining results.

Interpreting the Results

Live Metrics

Metric Meaning What to Expect
Sent Number of messages successfully sent to your input queue. Should match the message count you chose.
Received Number of result messages retrieved from the results queue. Should eventually equal Sent. If it stays at 0, your Lambda may have an error.
Avg Latency Average time (ms) between when the test started and when each result was processed. Includes Lambda cold start time. First batch will be slower.
Results/sec How fast results are arriving back, measured from the start of the test. Increases as Lambda scales up to handle the burst.

Results Table

Column Meaning
Opportunity ID First 8 characters of the UUID. Matches the message you sent.
Winner The advertiser ID your scoring function chose. This is the bid with the highest quality-adjusted score (not necessarily the highest raw bid).
Bid ($) The raw bid amount of the winner (in dollars CPM).
Score The quality-adjusted score of the winner: bid_amount * relevance * time_boost * device_weight.
Margin How much the winner's score exceeded the second-place score. A small margin means the auction was competitive.
Processed At The UTC timestamp when your Lambda finished processing this message. Useful for measuring real processing time.

Troubleshooting

Symptom Likely Cause Fix
Test Connection fails Expired or incorrect credentials Refresh your Learner Lab session and copy fresh keys
ERROR resolving queue URLs Stack not deployed, or student ID mismatch Run sam deploy --guided and verify your student ID matches
Sent goes up but Received stays at 0 Lambda is crashing or timing out Check CloudWatch Logs: aws logs tail /aws/lambda/adflow-YOURID-worker --follow
Received is less than Sent Some messages caused errors in your Lambda Check logs for ERROR entries. Failed messages go to partial batch failure and will be retried by SQS. Click "Poll Results Only" to pick up stragglers.
Results look wrong (wrong winner or score) Bug in compute_score or select_winner Run the unit tests again locally. Compare your output with the worked example in the assignment document.

Cleanup Between Test Runs

The yellow Cleanup section at the bottom lets you reset your environment between test runs. This is important because leftover messages from a previous run will mix with new ones, and duplicate records in DynamoDB will overwrite each other.

Button What It Does
Purge Input Queue Removes all messages from the input queue. SQS allows one purge per 60 seconds.
Purge Results Queue Removes all messages from the results queue.
Recreate DynamoDB Table Deletes and recreates the table (takes 30-60 seconds). This is the only way to delete all items from a DynamoDB table without scanning and deleting one by one.
Full Cleanup (All) Runs all three cleanups sequentially and resets the metrics display.

Activity Log

The dark terminal area at the very bottom shows timestamped entries for every action the tool performs. If something goes wrong, this is the first place to look. Error messages from AWS will appear here with the specific error code.

Cost Safety Reminder

  • SQS: first 1 million requests per month are free. A 500-message test uses about 100 API calls.
  • Lambda: first 1 million invocations per month are free. 500 messages = 500 invocations.
  • DynamoDB: on-demand pricing is negligible for 500 writes.
  • When you are done with the project, run sam delete --stack-name adflow-YOURID to remove all resources.
