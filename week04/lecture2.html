Part 1: Throughput vs Latency - Understanding Performance Metrics

Two Different Ways to Measure Performance

When we talk about system performance, we're usually talking about one of two things: throughput or latency. Understanding the difference is critical for designing scalable systems.

Metric Definition Units Example
Latency How long it takes to complete a single request Milliseconds (ms) API response time: 150ms
Throughput How many requests can be processed per unit time Requests per second (RPS) API can handle 10,000 RPS

The Highway Analogy

Think of a highway system:

Concept Highway Analogy System Analogy
Latency How long it takes one car to get from A to B How long one request takes to complete
Throughput How many cars can travel the highway per hour How many requests the system can handle per second
Improve Latency Increase speed limit, reduce traffic lights Optimize code, use caching, faster database queries
Improve Throughput Add more lanes to the highway Add more servers, horizontal scaling

Key Insight: They're Not Always Correlated

You can have high throughput with high latency (many slow requests), or low throughput with low latency (few fast requests). A 10-lane highway in rush hour traffic has high throughput but high latency. A single-lane country road at 3 AM has low throughput but low latency.

Tail Latency: Why Averages Lie

When measuring latency, the average (mean) can be misleading. What really matters is the tail latency - the worst-case performance that a small percentage of users experience.

Consider this scenario:

Request Latency
Requests 1-95 50ms each
Requests 96-99 500ms each
Request 100 5000ms (5 seconds!)
Average Latency 115ms - looks great!

The average is 115ms, which looks good. But 5% of your users are experiencing 500ms+ latency, and 1% are waiting 5 seconds. That's a terrible user experience!

Percentiles: A Better Way to Measure Latency

Instead of averages, we use percentiles to understand the distribution of latency:

Percentile What It Means Why It Matters
p50 (median) 50% of requests are faster than this Typical user experience
p95 95% of requests are faster than this Most users have this experience or better
p99 99% of requests are faster than this Catches outliers that affect 1 in 100 users
p99.9 99.9% of requests are faster than this Critical for high-scale systems (1 in 1000 users)

Why p99 Matters at Scale

If you have 1 million users per day and your p99 latency is 5 seconds, that means 10,000 users per day are experiencing 5+ second delays. That's a lot of frustrated users!

At Amazon scale, even p99.9 isn't enough - they care about p99.99 because even 0.01% of their traffic represents thousands of customers.

Thinking Ahead: The Distribution Problem

Now that you understand throughput vs latency and tail latency, let's think about a practical problem that brings all these concepts together.

A Thought Experiment: Pizza Delivery Service

Imagine you're building an online pizza ordering system. When a customer places an order through your website, your system needs to:

  • Accept the order and payment
  • Send the order to the kitchen
  • Track the order status (received, preparing, baking, out for delivery)
  • Send SMS notifications to the customer
  • Update the delivery driver's app
  • Record the order in the database for analytics

Here's the challenge: Making a pizza takes 15 minutes, but customers expect to place their order and get confirmation in under 2 seconds. During Friday dinner rush (6-8 PM), you get 200 orders per hour.

The Constraints

Order placement time: Must respond in under 2 seconds
Pizza preparation time: 15 minutes per pizza
Peak load (Friday 6-8 PM): 200 orders per hour (3-4 per minute)
Tasks per order: 6 different operations

Questions to Consider

Think about these questions before our next lecture:

  1. Horizontal Scaling: If your website runs on one server and gets slow during dinner rush, can you just add more servers? How would that help?
  2. Amdahl's Law: Which parts of taking an order can be parallelized? Which parts must be sequential? What's your bottleneck?
  3. Throughput vs Latency: How do you accept 200 orders per hour (throughput) while responding to each customer in under 2 seconds (latency)?
  4. The Coordination Problem: When a customer places an order, how does the website communicate with the kitchen, the SMS service, and the driver app? Do they all need to respond before you can confirm the order?
  5. Failure Handling: What happens if the SMS service is down? Should the customer's order be rejected? Or should you accept the order and retry the SMS later?
  6. Load Spikes: What if the local football team wins and suddenly 500 people order pizza at the same time? How do you prevent your system from crashing?

The Real Challenge: Temporal Decoupling

Notice that the problem isn't just about having enough servers. Even with unlimited capacity, you still face a fundamental challenge:

The order-taking service and the pizza-making service operate at different speeds.

Orders come in fast (200 per hour in bursts), but making pizzas is slow (15 minutes each). The website can't just wait for the pizza to be made before responding to the customer - that would tie up the web server for 15 minutes and create a terrible user experience.

You need a way to:

  • Accept orders quickly (respond to customer in under 2 seconds)
  • Process orders slowly (take 15 minutes to make the pizza)
  • Decouple the two (website doesn't wait for kitchen)
  • Distribute work across multiple kitchen stations
  • Handle failures gracefully (retry SMS if it fails, don't lose the order)
  • Coordinate multiple steps (kitchen, SMS, driver app, database)

The Missing Piece

All the concepts we've covered today - horizontal scaling, Amdahl's Law, throughput vs latency - are necessary but not sufficient. We need a mechanism to:

  • Store orders that need to be processed (without blocking the customer)
  • Distribute those orders across multiple kitchen stations
  • Ensure orders don't get lost if a system crashes
  • Allow different services (kitchen, SMS, driver app) to work at their own pace

What kind of system architecture would enable this?

Hint: Think About Real-World Analogies

Real-World System How It Handles Temporal Decoupling
Pizza Restaurant Cashier takes order quickly, kitchen makes pizza slowly. Orders are written on tickets and hung up for cooks to grab.
Coffee Shop Barista writes your name on a cup, you pay and step aside. Multiple baristas make drinks from the queue of cups.
Emergency Room Triage nurse sees you immediately, but you wait in queue based on severity. Doctors process patients as they become available.
Print Shop You submit print job instantly, but it sits in a queue. Multiple printers work through the queue in parallel.

What do all these systems have in common? They use a queue to buffer work between fast producers (taking orders) and slow consumers (making pizzas).

Part 2: The Problem - One Server Isn't Enough

The Pizza Ordering Challenge

Let's revisit our pizza ordering scenario with specific numbers:

  • Normal traffic: 50 orders per hour (website handles this fine)
  • Friday dinner rush: 200 orders per hour (website slows down)
  • Football game ends: 500 orders in 10 minutes (website crashes)

Your website is currently running on a single EC2 instance. When too many customers try to order at once, the server gets overwhelmed.

Solution 1: Vertical Scaling (The Wrong Answer)

Attempt 1: Buy a Bigger Server

You could upgrade from a t3.small to a t3.2xlarge (8x more CPU and RAM). This helps, but:

  • Expensive: Costs 8x more, even during slow hours
  • Limited: Eventually you hit the biggest instance size
  • Single point of failure: If that one server crashes, your entire website is down
  • Wasteful: You're paying for peak capacity 24/7, even at 3 AM when nobody orders pizza

Solution 2: Horizontal Scaling (The Right Answer)

Instead of one big server, use many small servers. During dinner rush, run 10 servers. At 3 AM, run 2 servers. This is horizontal scaling.

Application Load Balancer distributing traffic across multiple servers for horizontal scaling (Source: AWS)

Benefits of Horizontal Scaling

  • Cost-effective: Pay only for what you need, when you need it
  • Unlimited: Can scale to hundreds or thousands of servers
  • Resilient: If one server crashes, the others keep running
  • Flexible: Automatically add/remove servers based on traffic

But horizontal scaling introduces a new problem: How do customers know which server to connect to?

Part 3: Load Balancers - Traffic Distribution

What is a Load Balancer?

A load balancer sits in front of your servers and distributes incoming requests across all available servers. Think of it like a restaurant host who seats customers at different tables to balance the workload across waiters.

Without Load Balancer With Load Balancer
Customers connect to pizza.com (points to one server) Customers connect to pizza.com (points to load balancer)
All traffic goes to that one server Load balancer distributes traffic across 10 servers
If server crashes, website is down If one server crashes, load balancer routes traffic to the other 9

AWS Application Load Balancer (ALB)

AWS provides Application Load Balancer (ALB), a fully managed load balancer that:

  • Distributes traffic across multiple EC2 instances
  • Health checks - automatically stops sending traffic to unhealthy servers
  • SSL termination - handles HTTPS encryption for you
  • Path-based routing - send /api requests to one set of servers, /images to another
  • Sticky sessions - keep a user on the same server (if needed)

How Health Checks Work

The load balancer periodically sends a request to each server (e.g., GET /health). If the server responds with HTTP 200, it's healthy. If it fails 3 times in a row, the load balancer stops sending traffic to it.

Server Health Check Result Action
Server 1 HTTP 200 OK Receives traffic
Server 2 HTTP 200 OK Receives traffic
Server 3 Timeout (crashed) Removed from rotation

Part 4: Auto Scaling Groups - Dynamic Capacity

The Problem: How Many Servers Do You Need?

You have a load balancer, but you still need to decide: how many servers should be running?

  • Tuesday at 3 PM: 20 orders/hour - need 2 servers
  • Friday at 7 PM: 200 orders/hour - need 10 servers
  • Football game ends: 500 orders in 10 minutes - need 25 servers

You could manually start/stop servers, but that's tedious and slow. By the time you notice the traffic spike and start new servers, customers are already experiencing slowness.

Auto Scaling Groups (ASG)

Auto Scaling Group managing EC2 instances across availability zones (Source: AWS Documentation)

An Auto Scaling Group automatically adjusts the number of EC2 instances based on demand. You define:

  • Minimum capacity: Always run at least 2 servers (for redundancy)
  • Maximum capacity: Never run more than 50 servers (cost limit)
  • Desired capacity: Target number of servers (ASG adjusts this automatically)
  • Scaling policy: When to add/remove servers

Scaling Policies

Policy Type How It Works Example
Target Tracking Maintain a metric at a target value Keep average CPU at 50%
Step Scaling Add/remove instances based on thresholds If CPU > 70%, add 2 servers
Scheduled Scaling Scale at specific times Scale up Fri-Sun 5-9 PM

Example: Target Tracking Policy

Let's say you configure: "Keep average CPU utilization at 50%"

Time Traffic CPU Usage ASG Action Servers
3:00 PM Low 30% Remove 1 server 2
6:00 PM Rising 75% Add 2 servers 4
7:00 PM Peak 80% Add 4 servers 8
9:00 PM Declining 35% Remove 3 servers 5

Connecting to Amdahl's Law

Remember from Lecture 1: Amdahl's Law says your speedup is limited by the sequential portion of your code. For web applications, the key to horizontal scaling is making your servers stateless:

  • Session data: Store in DynamoDB or ElastiCache (not on the server)
  • User uploads: Save to S3 (not local disk)
  • No sticky sessions: Any server can handle any request

With stateless servers, you can scale from 2 to 100 instances with near-linear speedup!

Putting It Together: ALB + ASG

Here's how ALB and ASG work together:

  1. Customer visits pizza.com (DNS points to ALB)
  2. ALB receives the request and picks a healthy server from the ASG
  3. Server processes the request and returns the response
  4. If CPU usage rises above 50%, ASG launches new servers
  5. ALB automatically starts sending traffic to the new servers
  6. When traffic drops, ASG terminates excess servers

Great! We Solved Scaling... Or Did We?

ALB + ASG solves the problem of scaling your website (the part that accepts orders). But remember from Lecture 1: the website operates at a different speed than the kitchen.

Even with 100 web servers, you still have the temporal decoupling problem: orders come in fast, but processing (making pizzas, sending SMS, updating driver app) is slow.

We need a way to decouple the fast part (accepting orders) from the slow part (processing orders).

Part 5: Message Queues - Temporal Decoupling

The Queue Concept

Remember from Lecture 1: real-world systems like restaurants use queues to handle temporal decoupling. In a pizza restaurant:

  1. Cashier takes order and writes it on a ticket (fast - 30 seconds)
  2. Ticket is hung on a rack (the queue)
  3. Cook grabs next ticket when ready (slow - 15 minutes to make pizza)
  4. Customer gets confirmation immediately, doesn't wait for pizza to be made

We need the same pattern for our website!

Amazon SQS - Simple Queue Service

Amazon SQS is a fully managed message queue service. It acts as a buffer between your fast web servers and your slow processing workers.

Amazon SQS basic architecture with producers, queue, and consumers (Source: AWS Documentation)

Feature Description Why It Matters
Fully Managed No servers to manage, scales automatically Focus on business logic, not infrastructure
Unlimited Throughput Can handle millions of messages per second Handles traffic spikes automatically
Message Retention 1 minute to 14 days (default 4 days) Messages won't be lost if workers are down
At-Least-Once Delivery Messages may be delivered more than once Your code must be idempotent
Visibility Timeout Message hidden while being processed Prevents duplicate processing

How SQS Works: The Pizza Order Flow

Step Component What Happens Time
1. Order Website (Lambda) Customer submits order, Lambda sends message to SQS < 1 second
2. Queue SQS Message sits in queue until worker is ready Variable
3. Process Worker (Lambda) Worker pulls message, processes order, sends SMS 15 minutes
4. Delete Worker (Lambda) Worker deletes message from queue (success!) < 1 second

Key Insight: Throughput vs Latency

The website responds in < 1 second (great latency for the customer), while processing happens asynchronously over 15 minutes. With multiple workers pulling from the queue, you can process 200 orders per hour (high throughput).

This is exactly what we discussed in Lecture 1: optimizing for both latency AND throughput!

Visibility Timeout: Preventing Duplicate Work

When a worker pulls a message from SQS, the message becomes invisible to other workers for a set period (visibility timeout). This prevents two workers from processing the same order.

Scenario What Happens
Worker processes successfully Worker deletes message, it's gone forever
Worker crashes mid-processing After visibility timeout expires, message becomes visible again for another worker to try
Processing takes longer than timeout Message becomes visible again, another worker might process it (need idempotency!)

Important: Idempotency

Because SQS guarantees at-least-once delivery, your worker might receive the same message twice. Make sure your code is idempotent (safe to run multiple times):

  • Check if order has already been processed (query DynamoDB)
  • Use unique order IDs to prevent duplicate charges
  • Design operations to be safe when repeated

Part 6: Building the Pizza Ordering Demo

Architecture Overview

Let's build a simple demo that shows queuing in action. We'll create:

Serverless architecture with Lambda, SQS, and DynamoDB for asynchronous processing (Source: AWS Compute Blog)

  1. Static HTML page - hosted on S3, lets customers submit pizza orders
  2. API Gateway + Lambda - accepts orders and sends them to SQS (fast response)

Step 2: The Order API (Fast Response)

order_api.py

import json
import boto3
import os
import uuid
from datetime import datetime

sqs = boto3.client('sqs')
dynamodb = boto3.resource('dynamodb')

QUEUE_URL = os.environ['QUEUE_URL']
TABLE_NAME = os.environ['TABLE_NAME']
table = dynamodb.Table(TABLE_NAME)

def handler(event, context):
    # Parse the order from request body
    body = json.loads(event['body'])
    pizza_type = body.get('pizza_type', 'Pepperoni')
    size = body.get('size', 'Large')
    customer_name = body.get('customer_name', 'Anonymous')

    # Generate unique order ID
    order_id = str(uuid.uuid4())

    # Save order to DynamoDB (status: pending)
    table.put_item(Item={
        'orderId': order_id,
        'pizzaType': pizza_type,
        'size': size,
        'customerName': customer_name,
        'status': 'pending',
        'createdAt': datetime.utcnow().isoformat()
    })

    # Send message to SQS
    sqs.send_message(
        QueueUrl=QUEUE_URL,
        MessageBody=json.dumps({
            'orderId': order_id,
            'pizzaType': pizza_type,
            'size': size,
            'customerName': customer_name
        })
    )

    # Return immediately (< 1 second!)
    return {
        'statusCode': 200,
        'headers': {
            'Access-Control-Allow-Origin': '*',
            'Content-Type': 'application/json'
        },
        'body': json.dumps({
            'message': 'Order received!',
            'orderId': order_id,
            'estimatedTime': '15-20 minutes'
        })
    }

Understanding the Order API Code

Let's break down what this code does line by line:

Code Section What It Does Why It Matters
sqs = boto3.client('sqs') Initialize SQS client outside handler Reused across invocations (Lambda container reuse)
QUEUE_URL = os.environ['QUEUE_URL'] Get queue URL from environment variable SAM injects this automatically - no hardcoding!
order_id = str(uuid.uuid4()) Generate unique ID for order Prevents duplicate orders, enables idempotency
table.put_item(...) Save order to DynamoDB with status "pending" Persists order immediately - won't be lost if worker crashes
sqs.send_message(...) Send message to SQS queue Triggers worker Lambda asynchronously
return {'statusCode': 200, ...} Return success response immediately Customer gets response in < 1 second (doesn't wait for pizza to cook!)

Key Insight: Temporal Decoupling

The API function completes in under 1 second. It doesn't wait for the pizza to be made (which could take 15 minutes). The SQS message acts as a "work ticket" that the worker will process later. This is temporal decoupling - separating fast operations (accepting orders) from slow operations (making pizzas).

Step 3: The Order Worker (Slow Processing)

order_worker.py

import json
import boto3
import os
import time
from datetime import datetime

dynamodb = boto3.resource('dynamodb')
TABLE_NAME = os.environ['TABLE_NAME']
table = dynamodb.Table(TABLE_NAME)

def handler(event, context):
    # Lambda automatically polls SQS and invokes this function
    for record in event['Records']:
        # Parse the message
        message = json.loads(record['body'])
        order_id = message['orderId']
        pizza_type = message['pizzaType']
        size = message['size']
        customer_name = message['customerName']

        print(f"Processing order {order_id} for {customer_name}")

        # Update status to "preparing"
        table.update_item(
            Key={'orderId': order_id},
            UpdateExpression='SET #status = :status, updatedAt = :time',
            ExpressionAttributeNames={'#status': 'status'},
            ExpressionAttributeValues={
                ':status': 'preparing',
                ':time': datetime.utcnow().isoformat()
            }
        )

        # Simulate making the pizza (this would be real work in production)
        # In reality, this might call external APIs, process data, etc.
        time.sleep(10)  # Simulate 10 seconds of work

        # Update status to "ready"
        table.update_item(
            Key={'orderId': order_id},
            UpdateExpression='SET #status = :status, completedAt = :time',
            ExpressionAttributeNames={'#status': 'status'},
            ExpressionAttributeValues={
                ':status': 'ready',
                ':time': datetime.utcnow().isoformat()
            }
        )

        print(f"Order {order_id} is ready!")

        # If this function completes successfully,
        # Lambda automatically deletes the message from SQS

    return {'statusCode': 200}

Understanding the Worker Code

The worker Lambda is triggered automatically by SQS. Let's understand the key concepts:

Code Section What It Does Why It Matters
for record in event['Records']: Loop through SQS messages Lambda can receive multiple messages in one invocation (BatchSize)
message = json.loads(record['body']) Parse the message body SQS stores messages as strings, need to parse JSON
table.update_item(...) Update order status in DynamoDB Customer can check status without waiting for completion
time.sleep(10) Simulate slow work (making pizza) In production, this would be real processing (API calls, data transformation, etc.)
return {'statusCode': 200} Signal success to Lambda Lambda automatically deletes message from SQS (won't be reprocessed)

Important: Automatic Message Deletion

If your handler completes successfully (no exceptions), Lambda automatically deletes the message from SQS. If your handler throws an exception, the message stays in the queue and will be retried. This is why idempotency is critical - your code must handle processing the same message multiple times safely.

Connecting to Week 3 Concepts

Notice we're using the same DynamoDB patterns from Week 3: update_item with UpdateExpression and ExpressionAttributeNames. The difference is we're now triggering this Lambda from SQS instead of API Gateway!

Part 7: Adding SNS for Notifications (Optional Enhancement)

The Problem: Notifying Multiple Systems

When an order is ready, you might want to:

  • Send SMS to customer
  • Send email receipt
  • Update driver app
  • Log to analytics system

You could have the worker Lambda call each service directly, but that creates tight coupling. If you add a new notification channel, you have to modify the worker code.

Amazon SNS - Simple Notification Service

Amazon SNS implements the pub/sub (publish-subscribe) pattern. The worker publishes one message to an SNS topic, and SNS fans it out to multiple subscribers.

Amazon SNS fanout pattern distributing messages to multiple subscribers (Source: AWS)

Pattern Use Case Example
SQS (Queue) One producer, one consumer (or competing consumers) Order processing - one worker processes each order
SNS (Pub/Sub) One producer, many consumers (all get the message) Order ready - notify SMS, email, driver app all at once
SNS + SQS Fan-out Reliable delivery to multiple systems SNS publishes to multiple SQS queues, each with its own worker

Adding SNS to Our Demo

Add this to your SAM template:

template.yaml (addition)

  # SNS Topic for order notifications
  OrderNotificationTopic:
    Type: AWS::SNS::Topic
    Properties:
      TopicName: pizza-order-notifications
      Subscription:
        - Protocol: email
          Endpoint: your-email@example.com  # Replace with your email

Then update the worker to publish to SNS when order is ready:

order_worker.py (addition)

sns = boto3.client('sns')
TOPIC_ARN = os.environ['TOPIC_ARN']

# After updating status to "ready"
sns.publish(
    TopicArn=TOPIC_ARN,
    Subject=f'Pizza Order {order_id} Ready!',
    Message=f'Hi {customer_name}, your {size} {pizza_type} pizza is ready!'
)

Best Practice: SNS + SQS Fan-out

For production systems, don't subscribe Lambda directly to SNS. Instead, use the fan-out pattern:

  1. Worker publishes to SNS topic
  2. SNS fans out to multiple SQS queues (one for SMS, one for email, one for analytics)
  3. Each queue has its own Lambda worker

This provides better reliability and allows each system to process at its own pace.

Part 8: Retries and Exponential Backoff

Connecting Back to Lecture 1

In Lecture 1, we discussed how distributed systems need to handle failures gracefully. When calling external services (SMS API, payment gateway, etc.), failures are inevitable:

  • Network timeouts
  • Service temporarily down
  • Rate limiting (too many requests)
  • Transient errors (database deadlock, etc.)

Exponential Backoff with Jitter

Instead of retrying immediately (which can overwhelm a struggling service), use exponential backoff: wait longer between each retry. Add jitter (randomness) to prevent thundering herd problem.

Exponential backoff pattern showing increasing delays between retries (Source: AWS Architecture Blog)

Retry Attempt Base Delay With Jitter (Random 0-100%)
1st retry 1 second 0.5 - 1.0 seconds
2nd retry 2 seconds 1.0 - 2.0 seconds
3rd retry 4 seconds 2.0 - 4.0 seconds
4th retry 8 seconds 4.0 - 8.0 seconds
5th retry 16 seconds 8.0 - 16.0 seconds

Python Implementation

Exponential Backoff with Jitter

import time
import random

def call_external_api_with_retry(max_retries=5):
    base_delay = 1  # Start with 1 second

    for attempt in range(max_retries):
        try:
            # Try to call the external API
            response = external_api.send_sms(...)
            return response  # Success!

        except Exception as e:
            if attempt == max_retries - 1:
                # Last attempt failed, give up
                raise e

            # Calculate delay with exponential backoff
            delay = base_delay * (2 ** attempt)

            # Add jitter (random 0-100% of delay)
            jittered_delay = delay * random.random()

            print(f"Attempt {attempt + 1} failed, retrying in {jittered_delay:.2f}s")
            time.sleep(jittered_delay)

AWS SDK Built-in Retries

Good news: The AWS SDK (boto3 for Python) already implements exponential backoff with jitter for you! When you call SQS, DynamoDB, SNS, etc., the SDK automatically retries on transient errors.

You only need to implement your own retry logic when calling external (non-AWS) APIs.
