End-to-End Distributed Data Pipeline with Predictive Intelligence

Design, build, and deploy a complete distributed data engineering system , from raw ingestion through ML-powered prediction , served behind a polished, publicly accessible web application.

Part 1 The Big Picture

1.1 What You Are Building

In Project 1 you deployed a single Lambda function inside a pipeline someone else designed. Project 2 inverts that: you design the entire system. You choose the data, define the pipeline, train a predictive model, and ship a live application that a stranger on the internet can reach and use.

The final product is a hosted web application that answers a predictive question grounded in real data. The question, the data source, the architecture, and the look-and-feel of the application are entirely up to you. Originality matters , this is not a tutorial clone.

The Core Deliverable

A publicly accessible URL that serves a working, visually compelling application backed by a distributed data pipeline and a trained ML model. You will hand the instructor a URL and a test script; the instructor will run the script and verify the system works end-to-end.

1.2 What "Distributed" Means Here

Your pipeline must meaningfully leverage distributed computing infrastructure , not as window dressing, but as a structural requirement of the scale or complexity of the work. At minimum, two of the following layers must involve distributed processing:

  • Ingestion: Data lands in S3 or a cloud object store from an external source (API, scrape, public dataset, IoT feed).
  • Transformation: Spark (Databricks), dbt, or Polars processes raw data into analysis-ready tables using a medallion architecture (bronze → silver → gold).
  • Storage: Cleaned data persists in Delta, Iceberg, Parquet, DynamoDB, or Snowflake , not local CSV files.
  • Serving: A serverless or cloud-hosted backend (Lambda, API Gateway, Databricks Model Serving, or equivalent) handles prediction requests.
Streaming Is Optional

Kafka and real-time streaming are not required for this project. If your use case benefits from a streaming layer, you may include it for bonus consideration, but a well-designed batch pipeline is the baseline expectation.

1.3 The Predictive Component

Every project must include a machine learning model that predicts something meaningful based on the data you collected. This is not a course in ML theory , the model itself can be straightforward , but it must be trained on your pipeline's output and served through the application.

You will use Databricks MLOps tooling (MLflow experiment tracking, model registry, and optionally AutoML) to train, log, and manage your model. If you have taken other courses that covered classification, regression, or time-series forecasting, draw on that knowledge to select an appropriate approach for your domain.

MLOps Coverage

We will cover Databricks MLflow, experiment tracking, and model registry in class during Weeks 13–14. You do not need to wait , the Databricks MLflow documentation and AutoML quickstart are excellent starting points if you want to begin early.

3.2 The Test Script

The test script is how the I verify your system works end-to-end. It must:

  1. Send at least one request to your application's prediction endpoint (or interact with the UI programmatically).
  2. Print the response (the prediction result) to the terminal.
  3. Exit with code 0 on success, non-zero on failure.
  4. Include a comment block at the top explaining what it does and any required environment setup (e.g., pip install requests).

The script should be self-contained and runnable with a single command:

# Example: a minimal test script
python test_project.py

# Or:
bash test_project.sh
Example Test Script Structure

Your test script might look like this (yours will be tailored to your project):

"""
Test Script for [Your Project Name]
====================================
Sends a sample prediction request to the live application.

Requirements: pip install requests

Usage: python test_project.py
"""
import requests
import sys

URL = "https://your-app-url.com/api/predict"

# Sample input matching your model's expected features
payload = {
    "feature_1": 42,
    "feature_2": "category_a",
    "feature_3": 7.5
}

resp = requests.post(URL, json=payload, timeout=30)
if resp.status_code != 200:
    print(f"FAIL: Got status {resp.status_code}")
    sys.exit(1)

result = resp.json()
print(f"Prediction: {result['prediction']}")
print(f"Confidence: {result.get('confidence', 'N/A')}")
print("PASS")
sys.exit(0)

3.3 The One-Page Write-Up

One page. Not two. Not "one and a half." One page, single-spaced, 11pt minimum font, reasonable margins. The write-up must cover:

  1. What it predicts , State the prediction question in one sentence.
  2. Data source , Where the data comes from, how much of it, and how it is refreshed (or if it is a one-time load).
  3. Pipeline architecture , A brief description (with a small diagram if space allows) of how data flows from source to model to UI. Name the specific services used.
  4. Model approach , What type of model, what features, and how it was trained. Include one key metric (accuracy, RMSE, F1, etc.).
  5. What you learned , One honest paragraph about what was harder than expected and what you would change if you rebuilt it.

Part 4 The UI , Make It Sizzle

4.1 What "Sizzle" Means

Your application must look and feel like something you would be proud to show at a job interview. It should not look like a homework assignment. This is your chance to demonstrate taste, craft, and attention to detail.

Specifically, the UI must:

  • Have a clear visual identity , a cohesive color palette, typography, and layout. Not browser defaults.
  • Communicate the prediction clearly , the user provides input and sees a result without confusion.
  • Include at least one visualization , a chart, graph, map, or animated element that adds meaning (not decoration).
  • Feel alive , loading states, transitions, hover effects, or micro-animations. Static walls of text fail this criterion.
Originality

The concept must be original. You may not replicate a tutorial, class example, or publicly available demo app. The idea, the data source, and the presentation must come from you. If two submissions share the same concept, both will be flagged for review.

4.2 Inspiration, Not Prescription

You could build:

  • A dashboard that predicts next-day demand for bike-share stations in your city, with a live map.
  • A restaurant review analyzer that predicts health inspection risk from Yelp review text.
  • A music recommendation engine that predicts listening mood from weather and time-of-day data.
  • A sports outcome predictor that ingests live stats and shows real-time win probabilities with animated gauges.
  • A housing market tool that predicts time-on-market for listings based on neighborhood features.
  • An air quality forecaster that ingests EPA sensor data and predicts AQI for the next 24 hours.

These are examples to spark ideas , do not copy them. Your project must reflect your own interests and domain knowledge.

Part 9 Frequently Asked Questions

Can I use a Kaggle dataset?

Yes, but the dataset must still flow through a distributed pipeline (S3 → Databricks → Delta, not just pd.read_csv("local_file.csv")). The ingestion step can be a one-time upload to S3, but the subsequent transformation and storage must use cloud infrastructure.

What if my model is not very accurate?

That is fine. The grading focuses on whether the model is appropriate for the task, properly tracked in MLflow, and served through the pipeline , not on achieving state-of-the-art accuracy. Report your metrics honestly. A thoughtful model that gets 68% accuracy with clear feature engineering is worth more than a black-box model that claims 95% with no explanation.

Can I use Streamlit or Gradio instead of building a custom frontend?

Yes. Streamlit, Gradio, and similar frameworks count as your UI layer. However, the "sizzle" bar still applies , a raw Streamlit app with default styling and no visual thought will score lower on the UI category than one with custom theming, thoughtful layout, and polished interactions.

Do I need to keep the application running 24/7?

No. It must be live during the grading window (final exam week, May 18–22). Outside of that window, it can be offline. If your hosting has cold-start delays (e.g., free-tier Streamlit Cloud or Hugging Face Spaces), note that in your README and test script comments.

Can I use a technology we did not cover in class?

You may add technologies beyond what we covered (e.g., a React frontend, a PostgreSQL database, a Prefect orchestrator), but the core distributed pipeline must use at least two technologies from the course. Adding external tools should enhance your project, not replace the class content.

What if I am unsure whether my idea is ambitious enough?

Talk to the instructor. That is what Milestone 1 is for. A well-scoped project that works end-to-end is always better than an overly ambitious project that is half-broken at submission.
