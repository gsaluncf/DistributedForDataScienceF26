Event Streaming and Message Brokers

How Kafka-style brokers decouple producers from consumers, why teams adopt them, and the distributed systems complexities they introduce.

Part 1 Where Brokers Live in the Stack

1.1 Request/Response vs Event-Driven

Every distributed system you have built so far in this course uses the request/response pattern. A client sends a request to a server, the server processes it, and the client waits for the response. API Gateway + Lambda, DynamoDB queries, Snowflake SQL — all request/response.

This pattern works well when the caller needs the answer right now. But many real-world workflows do not. When a temperature sensor emits a reading, it does not wait for someone to acknowledge it. When a user clicks "purchase," the payment service does not need to wait for the recommendation engine to update its model. These are events — facts about something that happened — and they flow better through a different architecture.

Left: synchronous request/response. Right: asynchronous event streaming through a broker.

1.2 The Broker as Intermediary

A message broker sits between producers and consumers. Its job is to receive messages, store them durably, and deliver them to interested consumers. The broker decouples the two sides in three critical dimensions:

Dimension Without Broker With Broker
Time Producer and consumer must be online simultaneously Messages persist; consumer can read hours or days later
Identity Producer must know the address of every consumer Producer publishes to a topic; has no idea who reads it
Rate Consumer must process at the rate the producer sends Broker absorbs bursts; consumer pulls at its own pace
Where You Have Seen This Before

SQS in Project 1 was a message broker. Lambda polled the queue, processed messages, and deleted them. Kafka is the same fundamental pattern — producer, broker, consumer — but at a vastly different scale, with durable storage, replay, and multi-consumer support built in.

Part 2 What People Use Kafka For

2.1 Industry Use Cases

Use Case How It Works Who Does This
Event Sourcing Every state change is recorded as an immutable event in a topic. The current state is derived by replaying the event log. Financial systems, audit trails
Change Data Capture Database row-level changes (INSERT, UPDATE, DELETE) are streamed to a topic so downstream systems stay in sync without polling. Debezium + Kafka, data lake ingestion
ETL Decoupling Producers emit raw data to topics; multiple consumers transform and load it independently — a data warehouse, a search index, an ML feature store. LinkedIn, Uber, Netflix
Microservice Integration Services publish domain events ("OrderPlaced," "PaymentProcessed") instead of calling each other directly. This eliminates cascading failures. Any large platform with many services
IoT Telemetry Thousands of sensors emit readings continuously. A topic absorbs the firehose; consumers aggregate, alert, or archive. Smart factories, autonomous vehicles
Log Aggregation Application logs stream to Kafka, then fan out to monitoring (Datadog), search (Elasticsearch), and archival (S3). Every large-scale platform
The Common Thread

In every case above, the producer does not need to know who will read the data, how many consumers exist, or when they will read it. The broker absorbs the complexity. This is the fundamental value proposition of event streaming.

2.2 SQL Polling vs Event Streaming

Before Kafka, the most common way to move data between systems was to poll a database table. A job runs every N minutes, queries for new rows (using a WHERE updated_at > last_checkpoint filter), and copies them somewhere else. This works, but has structural weaknesses:

SQL Polling

-- Run every 5 minutes via cron
SELECT *
FROM   orders
WHERE  updated_at > :last_run
ORDER  BY updated_at;

-- Problems:
-- 1. 5-min latency minimum
-- 2. Clock skew can miss rows
-- 3. Deletes are invisible
-- 4. Polling load on the DB
-- 5. One consumer per query

Event Stream

# Runs continuously
for msg in consumer.poll():
    order = json.loads(msg.value)
    process(order)

# Advantages:
# 1. Sub-second latency
# 2. Guaranteed delivery
# 3. Deletes are explicit events
# 4. Zero load on source DB
# 5. N consumers, one stream

Part 3 Core Concepts

3.1 Topics, Partitions, and Offsets

A topic is a named, append-only log of messages. Think of it as a table, but you can only INSERT — never UPDATE or DELETE. Each message gets a sequential offset number within its partition.

A topic is split into one or more partitions. Each partition is an independent, ordered log stored on a single broker. Partitions are the unit of parallelism: more partitions means more consumers can read simultaneously.

Each partition is an independent ordered log. The message key determines which partition receives the message.

3.2 Consumer Groups

A consumer group is a set of consumers that cooperate to read a topic. The broker assigns each partition to exactly one consumer in the group. This means:

  • 1 consumer, 3 partitions: that consumer reads all three.
  • 3 consumers, 3 partitions: each consumer gets exactly one — perfect parallelism.
  • 5 consumers, 3 partitions: two consumers sit idle (you cannot have more consumers than partitions in a group).
The Key Insight

Consumer groups give you horizontal scalability on the read side. Need more throughput? Add consumers (up to the partition count). The broker automatically rebalances partition assignments — no code changes, no redeployment.

3.3 Brokers and Replication

In production, a Kafka cluster has multiple brokers (servers). Each partition has one leader and zero or more followers (replicas). Producers write to the leader; followers replicate the data. If the leader fails, a follower is promoted. This is the same leader-election pattern you studied in the P2P module.

Part 4 Complexities Streaming Introduces

4.1 The Five Hard Problems

Problem What Goes Wrong How Kafka Addresses It
Ordering Messages arrive out of order if spread across partitions Ordering is guaranteed within a partition. Use the message key to route related messages to the same partition.
Exactly-Once Delivery Network failures can cause duplicates (producer retries) or data loss (consumer commits offset before processing) Idempotent producers + transactional consumers. This is opt-in and has a throughput cost.
Backpressure Producer sends faster than consumer can process. Messages pile up; consumer falls behind ("consumer lag"). Kafka stores messages on disk. Consumer lag is a metric, not a failure. Add consumers, increase partitions, or batch processing.
Schema Evolution Producer changes the message format; consumer breaks because it expects the old format. Schema Registry (Avro, Protobuf, JSON Schema). Enforces backward/forward compatibility rules before a message is accepted.
Retention vs Cost Storing every message forever is expensive. Deleting too early breaks consumers that replay. Configurable retention (time-based or size-based). Common pattern: short retention on Kafka + long-term archival to S3/Delta Lake.

4.2 At-Least-Once vs At-Most-Once vs Exactly-Once

This is the single most discussed topic in streaming systems. Every team must choose a delivery guarantee based on their tolerance for duplicates versus data loss:

Guarantee Behavior When to Use
At-most-once Commit offset before processing. If processing fails, the message is lost. No duplicates. Metrics, logs, telemetry — where losing a few readings is acceptable.
At-least-once Process first, then commit offset. If commit fails after processing, the message is reprocessed. Possible duplicates. Most production systems. Make your consumer idempotent (same processing twice produces same result).
Exactly-once Transactional processing: offset commit and state update happen atomically. Financial transactions, billing. Has the highest latency and complexity cost.
Critical Design Decision

In practice, most teams choose at-least-once delivery and make their consumers idempotent. This is the same pattern you used in Project 1: your Lambda could receive the same SQS message twice, and you designed it to produce the same DynamoDB result regardless. Exactly-once sounds ideal but comes with real performance and complexity costs.

Part 6 Where Kafka Fits in the Data Platform

6.1 The Modern Data Stack with Streaming

In our course so far, data has moved in batches: upload a file, run a Spark job, query a warehouse. Kafka adds a real-time lane alongside the batch lane. The two patterns are not mutually exclusive — they complement each other.

Kafka as the central nervous system: all sources publish events; both streaming and batch consumers read at their own cadence.

6.2 Connecting to What You Already Know

Course Tool Role in the Stack Kafka Connection
SQS (Project 1) Message queue (point-to-point) Kafka generalizes this: multi-consumer, replay, ordering.
Spark (Week 7) Batch processing engine Spark Structured Streaming can read Kafka topics as a DataFrame source.
