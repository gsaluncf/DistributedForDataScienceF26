Part 1 Create Your Confluent Cloud Cluster

Before You Begin

You need a Confluent Cloud account. The free trial provides $400 in credits for 30 days — more than enough for this lab. No credit card is required to sign up.

1

Sign up for Confluent Cloud

Go to confluent.cloud/signup and create an account with your school email.

2

Create an Environment

After logging in, click Environments in the left sidebar, then Add cloud environment. Name it ds-lab. Select the Essentials Stream Governance package (free with trial).

3

Create a Basic Cluster

Inside your environment, click Add cluster. Choose the Basic type (lowest cost, ideal for learning). Pick the cloud provider and region closest to you (e.g., AWS us-east-2). Name it greenhouse-lab and click Launch cluster.

4

Create a Topic

Inside your cluster, click Topics in the left sidebar, then Create topic. Name it greenhouse-sensors with 3 partitions. Leave the rest as defaults and click Create with defaults.

5

Create an API Key

  1. In the left sidebar, click API Keys (under Cluster Overview)
  2. Click Create key
  3. Choose My account scope (Global access)
  4. Click Next and then Download and continue
  5. Save the downloaded file — it contains your Key (username) and Secret (password)
6

Find Your Bootstrap Server

In the left sidebar, click Cluster Settings. Under Endpoints, copy the Bootstrap server value. It will look like:

pkc-xxxxx.us-east-2.aws.confluent.cloud:9092
Checkpoint

You now have three values: the Bootstrap server URL, your API Key, and your API Secret. You will use these in the next step to configure the Python scripts.

Part 2 Configure the Python Scripts

1

Install the Python client

pip install -r requirements.txt
2

Create your credentials file

Copy kafka_config_example.py to _kafka_config.py:

# Windows (PowerShell)
Copy-Item kafka_config_example.py _kafka_config.py

# macOS / Linux
cp kafka_config_example.py _kafka_config.py
3

Fill in your credentials

Open _kafka_config.py in your editor and replace the three placeholder values:

KAFKA_CONFIG = {
    "bootstrap.servers": "pkc-xxxxx.us-east-2.aws.confluent.cloud:9092",
    "security.protocol": "SASL_SSL",
    "sasl.mechanisms":   "PLAIN",
    "sasl.username":     "YOUR_API_KEY",       # ← paste your Key here
    "sasl.password":     "YOUR_API_SECRET",    # ← paste your Secret here
}
Security

The file _kafka_config.py starts with an underscore, so it is automatically excluded from git by this repository's .gitignore rules. Never commit credentials to version control.

Part 3 Exercise 1: Produce Messages

3.1 Run the producer

python producer.py --count 20

Watch the delivery reports. Each line shows the topic, partition number, and offset. Messages route to different partitions based on the sensor_id key.

3.2 Observe in the Confluent Console

Go back to the Confluent Cloud UI. Click Topics → greenhouse-sensors → Messages. You should see your 20 JSON messages spread across the 3 partitions.

3.3 Questions to answer

  1. Do messages from the same sensor_id always land in the same partition? Why?
  2. What happens if you run the producer again with --count 10? Do the new messages start at offset 0 or continue from the previous offset?
  3. Open producer.py and find the producer.produce() call. What would happen if you removed the key= argument? (Try it.)

Part 4 Exercise 2: Consume Messages

4.1 Start a single consumer

python consumer.py

The consumer reads from auto.offset.reset: earliest, so it replays all existing messages. Each line shows [partition:offset], sensor ID, temperature, humidity, and a running average.

4.2 Leave it running and produce more data

Open a second terminal and run:

python producer.py --count 10

Switch back to the consumer terminal. The new messages appear in near-real-time — even though the broker is in the cloud.

4.3 Questions to answer

  1. When the consumer first starts, does it read messages in global order or partition order?
  2. Stop the consumer (Ctrl-C) and restart it. Does it re-read all messages? Why or why not? (Hint: check the enable.auto.commit setting.)

Part 5 Exercise 3: Consumer Groups

5.1 Scale out with multiple consumers

Open three terminal windows. In each one, start a consumer with the same group:

# Terminal 1
python consumer.py --group greenhouse-team

# Terminal 2
python consumer.py --group greenhouse-team

# Terminal 3
python consumer.py --group greenhouse-team

Now produce a burst of messages from a fourth terminal:

python producer.py --count 60 --burst

5.2 Observe the partition assignment

Each consumer should receive messages from a different subset of partitions. With 3 partitions and 3 consumers, each consumer handles exactly one partition.

5.3 Observe rebalancing

Kill one of the three consumers (Ctrl-C). The broker detects the departure and rebalances: the remaining two consumers split three partitions between them. Produce another burst and observe the redistribution.

5.4 Questions to answer

  1. What happened when you stopped one consumer? How long did it take for the remaining consumers to pick up the orphaned partition?
  2. Start a fourth consumer in the same group. Does it receive any messages? Why?
  3. Start a consumer with a different group name (--group analytics-team). Does it receive messages that were already consumed by greenhouse-team? Why?
Key Takeaway

Consumer groups are the mechanism for horizontal read scaling in Kafka. Each group independently tracks its own offsets. Adding a new group does not affect existing groups — both see the full topic. Within a group, partitions are divided so no message is processed twice.

Part 6 Exercise 4: Extend the Pipeline

6.1 Add an alert threshold

Modify consumer.py to print a warning when any sensor reports a temperature above 30.0 C:

6.2 Add a new sensor type

Modify producer.py to include a light_lux field in each reading (a random float between 200 and 8000). Update the consumer to display it. The old consumer code still works even though the message format changed — this is forward compatibility via optional fields.

6.3 Create a second topic and produce alerts

In the Confluent Cloud UI, create a new topic called greenhouse-alerts with 1 partition. When the consumer detects a high temperature, have it produce an alert message to greenhouse-alerts. This creates a two-stage streaming pipeline: raw telemetry → alert generation.

Part 7 Stretch: Kafka to Delta Live Tables

In a production data platform, you rarely consume Kafka with standalone Python scripts. A streaming engine like Spark Structured Streaming reads the topic and lands the data as a Delta table. Databricks DLT makes this declarative:

import dlt
from pyspark.sql.functions import col, from_json, schema_of_json

SAMPLE = '{"sensor_id":"s","temperature_c":0.0,"humidity_pct":0.0,"timestamp":"t"}'

@dlt.table(
    comment="Raw sensor readings ingested from Kafka",
    table_properties={"quality": "bronze"},
)
def bronze_sensors():
    return (
        spark.readStream
            .format("kafka")
            .option("kafka.bootstrap.servers", "pkc-xxxxx.confluent.cloud:9092")
            .option("kafka.security.protocol", "SASL_SSL")
            .option("kafka.sasl.mechanism", "PLAIN")
            .option("kafka.sasl.jaas.config",
                    "...PlainLoginModule required username='KEY' password='SECRET';")
            .option("subscribe", "greenhouse-sensors")
            .option("startingOffsets", "earliest")
            .load()
            .withColumn("parsed", from_json(col("value").cast("string"),
                                            schema_of_json(SAMPLE)))
            .select("parsed.*", "topic", "partition", "offset", "timestamp")
    )
Free-Tier Limitation

Databricks Community Edition does not support DLT pipelines. If you have access to a paid Databricks workspace, you can run this directly. Otherwise, treat this as a read-and-understand exercise — the pattern is what matters.

Once the Bronze table exists as a streaming Delta table, dbt can model it downstream with standard SQL transformations:

Sensors --produce--> Confluent Cloud --readStream--> DLT (Bronze)
                                                          |
                                                      DLT (Silver) -- quality rules
                                                          |
                                                      DLT (Gold) -- aggregations
                                                          |
                                                      dbt models -- business logic SQL
