Unit 1 Foundations: Git and AWS Onboarding

What we covered

Git workflow (branch, commit, push, pull, merge, force push and why not), GitHub PR mechanics, merge conflicts. The AWS CLI, profiles, aws sts get-caller-identity as the canonical "who am I" check. The shape of an AWS account and the three pieces of credentials.

Flashcards
  • git push --force: rewrites remote history, breaks teammates' clones
  • Fast-forward vs merge commit: linear advance vs an explicit join point in the graph
  • Merge conflict: same lines edited on different branches, must resolve manually
  • IAM user vs IAM role: user has long-lived keys, role is assumed and gives short-lived credentials
  • AWS named profile: --profile name flag, lives in ~/.aws/credentials
  • Env vars: AWS_PROFILE, AWS_REGION, AWS_ACCESS_KEY_ID override the file
  • aws sts get-caller-identity: the canonical "who am I" check
  • Three credential parts: access key ID, secret access key, session token
  • .gitignore: the only thing standing between you and an exposed key on GitHub

Unit 2 S3, Cloud Computing, Distributed Systems

What we covered

S3 buckets and the global namespace, static hosting, the cannon trajectory simulator as a worked example of when you do and do not need a server, the difference between "client only", "client plus API", and "client plus API plus database".

Flashcards
  • S3 bucket names are globally unique across all AWS accounts
  • Static hosting: index.html + bucket policy, no server
  • Three architecture shapes: client only, client + API, client + API + DB
  • Cannon simulator: pure client-side compute, zero backend needed
  • Storage classes: Standard, Standard-IA, Glacier (cheaper to keep, slower to read)
  • Presigned URL: time-limited signed access to one object, no AWS credentials needed by the caller
  • CORS on the bucket is what lets a browser at a different origin actually upload or download
  • Bucket policy vs object ACL: coarse, JSON, the modern default vs per-object legacy
  • Region is the failure boundary, Availability Zone is the rack-level isolation inside it

Unit 3 Serverless: Lambda, DynamoDB, SAM

What we covered

The AWS Lambda execution model, cold starts vs warm starts, why boto3.resource(...) belongs at module scope. DynamoDB partition keys, hot partitions, get_item vs query vs scan, and the Decimal-not-float rule. SAM as a way to describe Lambda + queue + table in one YAML file.

Flashcards
  • boto3: the official AWS SDK for Python; every AWS API call you make from Python goes through it
  • boto3.client vs boto3.resource: low-level (every API parameter spelled out) vs higher-level object wrappers
  • Cold vs warm start: first invocation pays init cost, subsequent ones reuse the same container and its module-scope state
  • boto3.resource(...) at module scope: built once per container, reused across warm invocations, never rebuilt inside the handler
  • Why it matters: avoids paying TCP plus TLS plus auth on every call; cuts handler time and cost
  • What to watch out for: do not stash per-request user state in module-scope objects (the next caller in that container will see it), and the connection pool inside the client is finite
  • Lambda execution role: the IAM role Lambda assumes to call other AWS services on your behalf
  • Lambda timeout: default 3 seconds, hard maximum 15 minutes
  • Provisioned concurrency: pre-warmed containers to skip cold starts (costs money even when idle)
  • requirements.txt is packaged at build time, not pip-installed at invoke time
  • DynamoDB partition key: routes the row to a physical partition
  • Hot partition: too many requests to one key value, throttling
  • get_item (O(1) by key) vs query (key range) vs scan (full table, avoid)
  • Decimal not float: DynamoDB rejects floats
  • SAM: one YAML describes Lambda + queue + table + permissions; sam build then sam deploy --guided

Unit 4 Scaling Laws and Reliability

What we covered

Moore's Law and the clock-speed wall around 4 GHz. Amdahl's Law (the serial fraction caps your speedup) and Gustafson's Law (let the problem grow with the cores). The shift from "scale up" to "scale out" as a software-design consequence. Idempotency as the property that lets a system safely retry.

Flashcards
  • Moore's Law: transistor density doubles roughly every 2 years
  • Clock speed wall: physics caps single-core speed around 4 GHz
  • Amdahl's Law: the serial fraction is the ceiling on parallel speedup
  • Gustafson's Law: scale the problem with the cores, not the speedup of a fixed problem
  • Scale up vs scale out: bigger machine vs more machines
  • Latency vs throughput: time per request vs requests per second; you can trade one for the other
  • Strong vs eventual consistency: read sees the latest write, vs read may see a slightly stale value
  • CAP theorem: under a network partition you choose consistency or availability
  • Backpressure: slow the producer when the consumer cannot keep up
  • Idempotent: safe to retry, same result every time

Unit 5 Queues, Retries, the Distributed Cipher Lab

What we covered

SQS visibility timeout, why the same message can be delivered twice, how to design a worker so that double delivery does not produce double side effects. The Distributed Cipher lab as a place where all of this had to be real, not theoretical.

Flashcards
  • SQS visibility timeout: message hidden from other workers after delivery, reappears if not deleted in time
  • At-least-once delivery: the same message can arrive twice, must design for it
  • Idempotent worker: double delivery produces no extra side effects
  • Standard vs FIFO queue: high throughput, possible duplicates and reorder, vs ordered, deduplicated, lower throughput
  • Dead letter queue: catches messages that fail repeatedly; redrive policy says after how many tries
  • Batching: up to 10 messages per send or receive call to amortize the request overhead
  • Long polling: cheaper and faster than constant short polls
  • "Exactly once" is a fairy tale you build out of at-least-once plus an idempotent consumer

Unit 6 pandas to Polars

What we covered

Why pandas runs out of memory on a single laptop with multi-gigabyte files. Polars's lazy API, predicate pushdown, projection pushdown, and the Arrow columnar format underneath both Polars and Spark. The single-machine sweet spot before you have to reach for a cluster.

Flashcards
  • pandas: row-based, eager, single-threaded, runs out of memory around 1-2 GB
  • Polars: columnar, lazy, multi-threaded, scales to roughly RAM size
  • The GIL: the reason pure-Python loops in pandas cannot use more than one core
  • Vectorization: the trick that makes NumPy and Polars fast (C loops, not Python loops)
  • Predicate pushdown: apply filters at file scan, not after load
  • Projection pushdown: read only the columns you need
  • Apache Arrow: shared columnar in-memory format under both Polars and Spark
  • DuckDB: another single-machine analytics engine, SQL on top of Parquet and Arrow
  • Dask: the pandas-style API spread across many cores or many machines

Unit 7 Spark, the Lakehouse, and Unity Catalog

What we covered

Spark transformations are lazy and only execute when an action is called (the DAG, the Catalyst optimizer). Unity Catalog as "DNS for data" with its three-level namespace (catalog.schema.table). Why a columnar query on Parquet is so much faster than the same query on CSV. The Lakehouse pattern as the answer to "data swamps".

Flashcards
  • Lazy: transformations build a DAG, only actions trigger execution
  • Catalyst: Spark's query optimizer, rewrites the plan before it runs
  • Driver vs executor: the planner that holds your code vs the workers that run the tasks
  • Shuffle: the expensive operation when data has to move between executors
  • Broadcast join: small table sent to every executor, no shuffle needed
  • Unity Catalog three-level namespace: catalog.schema.table
  • Parquet vs CSV: columnar, compressed, schema-aware, much faster scans
  • Lakehouse: cheap object storage plus warehouse-grade governance and ACID
  • Delta Lake: the table format that gives you ACID transactions on top of Parquet
  • Time travel: VERSION AS OF on a Delta table reads it as it was at that commit
  • Iceberg, Hudi: the other two open table formats in the same family as Delta

Unit 8 Medallion Architecture: Bronze, Silver, Gold

What we covered

Bronze as raw, append-only, all-strings, untouched. Silver as cleaned, typed, deduplicated, modeled. Gold as business-ready aggregates and joins. The custodian-report worked example. Why separating Silver from Gold pays off as soon as more than one team needs the cleaned data.

Flashcards
  • Bronze: raw, append-only, all strings, no transformations
  • Silver: cleaned, typed, deduplicated, modeled
  • Gold: business-ready aggregates and joins, what dashboards and reports read
  • Why split Silver from Gold: many Gold tables can share the same Silver source
  • Reprocess from Bronze if Silver logic has a bug, no need to re-ingest
  • Idempotent Bronze ingest: the same source file landed twice produces no duplicates downstream
  • SCD Type 1 vs Type 2: overwrite the row vs keep history with valid_from and valid_to
  • Star schema: a fact table in the middle, dimension tables hanging off it

Unit 9 Declarative Pipelines and dbt

What we covered

The imperative way: notebooks chained by Workflows, you wire the dependency graph by hand. The declarative way: @dlt.table() in Lakeflow Declarative Pipelines (DLT), where you describe the contract and the platform handles ordering, retries, and quality expectations. dbt as the SQL-first cousin: every model is a SELECT, ref() and source() build the DAG, the warehouse does the work.

Flashcards
  • Imperative pipeline: notebooks chained by Workflows, you wire the DAG manually
  • Declarative pipeline: @dlt.table() describes the contract, platform handles ordering and retries
  • DLT expectations: data quality rules enforced at write time
  • Streaming table vs materialized view in DLT: append-only incremental vs full recompute on a schedule
  • dbt model: every model is a SELECT, ref() and source() build the DAG
  • Materializations: view, table, incremental, ephemeral
  • Incremental needs a watermark column (typically updated_at) so dbt knows what is new
  • Jinja in dbt: SQL with conditionals, loops, and macros
  • dbt snapshot: the built-in SCD2 helper
  • dbt tests: unique, not_null, accepted_values, custom SQL

Unit 10 MLOps

What we covered

The dev / staging / production framework. MLflow as the experiment tracker (params, metrics, artifacts, every run logged). The Model Registry as named, versioned models with stage transitions. The "imagine if all the engineers in the world suddenly lost GitHub" framing for why this matters in data science.

Flashcards
  • Three stages: dev, staging, production
  • MLflow tracks: parameters, metrics, artifacts, every run
  • Model Registry: named, versioned models with stage transitions
  • Reproducibility: log enough that you can rerun the experiment six months later
  • Feature store: shared, versioned features used by both training and serving
  • Training-serving skew: the same feature computed two different ways in the two paths
  • Model drift vs data drift: the model degrades vs the inputs themselves shift
  • Canary or champion-challenger: route a small slice of traffic to the new version before full rollout
  • Pickle is a known security risk for distributing model files
  • "GitHub for models": versioning, lineage, rollback

Unit 11 Kafka and Event Streaming

What we covered

Topics, partitions, offsets. The producer key as the routing input (hash(key) mod num_partitions) and what that gives you (per-key ordering) and what it costs you (a hot key is a hot partition). Consumer groups, rebalancing, the greenhouse-sensor lab.

Flashcards
  • Topic, partition, offset are the three coordinates of a Kafka record
  • Producer key routes via hash(key) mod num_partitions
  • Same key always lands on the same partition (per-key ordering)
  • Hot key creates a hot partition (the cost of ordering)
  • Consumer group: each partition consumed by exactly one member at a time
  • Rebalancing: triggered when a consumer joins or leaves the group
  • Replication factor: how many brokers hold a copy of each partition
  • Schema Registry: Avro or Protobuf contract enforcement on the topic
  • Compaction vs retention: keep only the latest value per key vs delete after N days
  • Kafka Connect, Kafka Streams, ksqlDB are the rest of the broader Kafka ecosystem

Unit 12 Specialized Compute (Runpod, GPUs)

What we covered

Runpod serverless workers, the GPU vs CPU split, queue-based workers vs load-balanced HTTP endpoints. Compute as something you rent for ninety seconds at a time. The economics of "I do not own any of this hardware and I never will."

Flashcards
  • Serverless GPU: rent for roughly 90 seconds per request, billed by the second
  • Queue worker (async job) vs HTTP endpoint (sync request)
  • GPU vs CPU: massively parallel matrix math vs general-purpose compute
  • Cold start on GPU is real and expensive (model load time)
  • Quantization (int8, int4): smaller weights, lower memory, slight accuracy loss
  • Batching: pile multiple requests into one GPU call to amortize the per-call overhead
  • vLLM, TGI: inference servers that handle batching and KV-cache reuse for you
  • Container image size matters: bigger image, slower pull, slower cold start
  • You never own the hardware

Unit 15 You Can Now Do This

Most of you have not yet noticed how much you picked up. Read this list slowly. Every line is something you could not credibly do in January. Now you can.

Capability

You can deploy a multi-component distributed system. SAM template, Lambda worker, SQS input and results queues, DynamoDB table. You did this in Project 1.

Capability

You can reason about scaling tradeoffs. Vertical vs horizontal. Amdahl vs Gustafson. Hot partitions vs balanced ones. Idempotent vs unsafe-to-retry. You have the vocabulary now.

Capability

You can move from local to cluster to cloud. pandas on a laptop, Polars when pandas runs out, Spark on Databricks when Polars runs out, S3 plus Lambda when you need it to be live. You know the ladder.

Capability

You can choose between batch and streaming systems. Daily Medallion job vs Kafka topic with consumer groups. You have built both shapes.

Capability

You understand cost implications in real environments. Per-call Lambda pricing, GPU-minute Runpod pricing, storage-vs-compute separation, why a scan is the expensive query. You know what to watch so a runaway scan or a forgotten cluster does not catch you off guard.

Capability

You can wire an end-to-end pipeline with a model in it. Ingestion, processing, storage, training, registry, serving. Project 2 was the integration test of the whole semester and you shipped it.
