Unit 1 Foundations: Git and AWS Onboarding

What we covered

Git workflow (branch, commit, push, pull, merge, force push and why not), GitHub PR mechanics, merge conflicts. The AWS CLI, profiles, aws sts get-caller-identity as the canonical "who am I" check. The shape of an AWS account and the three pieces of credentials.

Flashcards
  • What is Git: distributed version control; tracks changes to files over time, every clone is a full copy of the history
  • Repository (repo): the folder Git tracks plus the hidden .git directory that holds the history
  • Commit: a snapshot of the repo at a point in time, with author, message, and a parent commit
  • Branch: a named pointer to a commit; lets parallel work happen without stepping on each other
  • Merge: combine two branches; produces a merge commit with two parents
  • Fast-forward vs merge commit: linear advance (no new commit) vs an explicit join point in the graph
  • Pull request (PR): a GitHub workflow for proposing and reviewing a merge before it lands on main
  • Merge conflict: same lines edited on different branches, must resolve manually
  • git push --force: rewrites remote history, breaks teammates' clones
  • .gitignore: list of paths Git refuses to track; the only thing standing between you and an exposed key on GitHub
  • What is AWS: Amazon Web Services, the largest public cloud; you rent compute, storage, networking, and managed services by the second
  • AWS account: a billing and isolation boundary identified by a 12-digit ID
  • Region: a geographic area such as us-east-1; the failure boundary
  • Availability Zone (AZ): a rack-level isolation inside a Region
  • What is IAM: Identity and Access Management; the AWS service that decides who can do what
  • IAM user: a long-lived identity with a username and access keys
  • IAM role: a short-lived identity that something assumes (a Lambda, an EC2 instance, a developer via SSO)
  • What is the AWS CLI: the command-line tool for talking to every AWS API from your terminal
  • Named profile: --profile name flag, lives in ~/.aws/credentials; lets one machine talk to many accounts
  • Env vars: AWS_PROFILE, AWS_REGION, AWS_ACCESS_KEY_ID override the file
  • Three credential parts: access key ID, secret access key, session token
  • aws sts get-caller-identity: the canonical "who am I" check

Unit 2 S3, Cloud Computing, Distributed Systems

What we covered

S3 buckets and the global namespace, static hosting, the cannon trajectory simulator as a worked example of when you do and do not need a server, the difference between "client only", "client plus API", and "client plus API plus database".

Flashcards
  • What is the cloud: someone else's data center, rented by the second; you stop owning hardware
  • Public cloud: AWS, Azure, GCP; anyone with a credit card can use them
  • Private cloud: the same idea, owned by one company, behind their firewall
  • Hybrid cloud: some workloads on public cloud, some on private, connected by a network
  • What is a distributed system: work and data spread across multiple machines connected by a network, presenting itself as one system
  • Why distribute: more capacity than any single machine and survival when one machine fails
  • The eight fallacies of distributed computing: the network is reliable (it is not), latency is zero (it is not), bandwidth is infinite (it is not), the network is secure, topology never changes, there is one administrator, transport cost is zero, the network is homogeneous
  • Centralized: one machine, one owner; your laptop, a single database server
  • Distributed: many machines, one owner or one operator; AWS, Google's data centers
  • Decentralized: many machines, many owners, no single coordinator; Bitcoin, BitTorrent, IPFS
  • Federated: many independent owners cooperating by protocol; email, Mastodon, Git itself
  • Region is the geographic failure boundary; an AZ is rack-level isolation inside it
  • Edge / CDN: cached copies of content close to users (CloudFront, Cloudflare); cuts latency, cuts origin load
  • SLA (Service Level Agreement): a contract about uptime, e.g. 99.9% means about 8.77 hours of allowed downtime per year
  • Durability: the chance of losing a byte; S3 advertises eleven nines (99.999999999%)
  • Availability: the chance the system answers when you ask; lower bar than durability
  • What is S3 (Simple Storage Service): AWS's object store; key plus bytes plus metadata at planetary scale
  • Object storage vs filesystem: flat key namespace, no folders, no inodes, no fseek; vs hierarchical directories and random access
  • S3 bucket: a top-level container; bucket names are globally unique across all AWS accounts
  • Static hosting: bucket holds index.html and assets, no server in the picture
  • Storage classes: Standard (hot), Standard-IA (infrequent), Glacier (archival); cheaper to keep, slower to read
  • What is CORS (Cross-Origin Resource Sharing): a browser-enforced rule that blocks one origin from reading another origin's response by default
  • Why CORS gets in your way: during local dev your front end is on localhost:3000 and your bucket is on s3.amazonaws.com, so the browser blocks the call until you configure the bucket
  • Why CORS exists anyway: it is the basic line of defense against a malicious site reading user data from your bucket through a victim's browser
  • Bucket policy vs object ACL: coarse JSON policy on the whole bucket (modern default) vs per-object legacy ACLs
  • Eventual vs strong consistency: read might see a stale value vs read always sees the latest write; S3 is now strong read-after-write
  • Architecture shape 1, client only: the browser does all the work; the cannon trajectory simulator was this
  • Architecture shape 2, client + API: the browser calls a backend for compute or shared logic
  • Architecture shape 3, client + API + database: the backend stores state across users and sessions
  • Architecture shape 4, client + API + DB + queue + worker: an asynchronous job runs in the background; Project 1 was this
  • Three-tier architecture: presentation, application logic, data; the classic web app layout
  • Stateless service: any instance can serve any request; easy to scale horizontally because there is no per-instance memory
  • Stateful service: a specific instance owns specific data; harder to scale, needs sticky routing or careful sharding
  • Monolith: one deployable artifact, one codebase; simple at small scale, painful past a certain size
  • Microservices: many small services, each with its own deployable; complex but independently scalable and independently owned
  • Cannon trajectory simulator: pure client-side compute, zero backend needed; the worked example for "do not add a server you do not need"
  • When you actually need a server: shared state across users, secrets that must not leak to the browser, heavy compute, scheduled work
  • When you actually need a database: durability, query, transactions, indexing

Unit 3 Serverless: Lambda, DynamoDB, SAM

What we covered

The AWS Lambda execution model, cold starts vs warm starts, why boto3.resource(...) belongs at module scope. DynamoDB partition keys, hot partitions, get_item vs query vs scan, and the Decimal-not-float rule. SAM as a way to describe Lambda + queue + table in one YAML file.

Flashcards
  • What is serverless: you bring code, the platform brings everything else (servers, scaling, patching); you pay per invocation, not per hour of idle
  • What is AWS Lambda: a managed service that runs your function in response to an event, with no server for you to provision or maintain
  • How Lambda works: you upload code; AWS spins up a small container when an event arrives; runs your handler; freezes or destroys the container after
  • Trigger / event source: what fires the function, e.g. an SQS message, an S3 put, an HTTP call via API Gateway, a CloudWatch schedule
  • Handler: the function AWS calls; typically def lambda_handler(event, context)
  • event: the JSON payload describing what triggered this invocation (queue message body, HTTP request, S3 key, etc.)
  • context: runtime metadata; request ID, log group name, milliseconds remaining
  • Memory parameter: 128 MB up to 10 GB; CPU scales linearly with memory (more memory means a faster CPU slice too)
  • Timeout: default 3 seconds, hard maximum 15 minutes
  • Execution role: the IAM role Lambda assumes to call other AWS services on your behalf
  • requirements.txt is packaged at build time, not pip-installed at invoke time; whatever you forgot is not there
  • Lambda layers: a way to share libraries or large binaries across functions without bundling them into every zip
  • Provisioned concurrency: pre-warmed containers to skip cold starts; costs money even when idle
  • Cold vs warm start: first invocation pays init cost (download, unzip, import); subsequent calls reuse the same container and its module-scope state
  • Module scope persists across warm invocations: anything outside the handler is built once per container and lives until that container is recycled
  • What is boto3: the official AWS SDK for Python; every AWS API call you make from Python goes through it
  • boto3.resource(...) at module scope: built once per container, reused across warm invocations, never rebuilt inside the handler
  • What to watch out for: do not stash per-request user state in module-scope objects, the next caller in that container will see it; and the connection pool inside the client is finite
  • What is DynamoDB: AWS's managed serverless NoSQL key-value and document store; single-digit-millisecond reads at any scale
  • Partition key: the field that decides which physical partition the row lives on
  • Sort key (optional): a second key that orders rows within a partition; lets you query a range
  • Hot partition: too many requests to one partition-key value, throttling
  • get_item (O(1) lookup by full key) vs query (range within a partition) vs scan (walks the whole table, avoid in production)
  • Decimal not float: DynamoDB rejects Python floats; use decimal.Decimal
  • What is SAM (Serverless Application Model): a YAML format that lets you describe Lambda + queue + table + permissions in one file; AWS converts it to CloudFormation under the hood
  • SAM workflow: sam build then sam deploy --guided are the two commands you actually run

Unit 4 Scaling Laws and Reliability

What we covered

Moore's Law and the clock-speed wall around 4 GHz. Amdahl's Law (the serial fraction caps your speedup) and Gustafson's Law (let the problem grow with the cores). The shift from "scale up" to "scale out" as a software-design consequence. Idempotency as the property that lets a system safely retry.

Flashcards
  • What is scaling: adding capacity to handle more work; can be vertical (scale up) or horizontal (scale out)
  • Scale up (vertical): bigger machine, more cores, more RAM, faster disk; simple but capped by what you can buy
  • Scale out (horizontal): more machines, each doing a slice of the work; harder to design but no upper bound
  • What is parallelism: doing multiple pieces of work at the same time on different cores or machines
  • Concurrency vs parallelism: dealing with many things at once vs actually doing many things at once
  • Moore's Law: transistor density on a chip roughly doubles every 18 to 24 months; the historical reason single-machine performance grew for decades
  • Clock-speed wall: physics caps single-core clock speed around 4 GHz; heat and power kill anything faster, which is why we got more cores instead of faster cores
  • Amdahl's Law: the serial fraction of a program is the ceiling on speedup, no matter how many cores you add; if 10% is serial, max speedup is 10x even with infinite cores
  • Gustafson's Law: in practice you do not run a fixed problem on more cores; you run a bigger problem in the same time, so parallel speedup compounds
  • Latency vs throughput: time per single request vs requests served per second; you can trade one for the other
  • Strong consistency: a read always sees the latest write
  • Eventual consistency: a read may see a slightly stale value, but the system converges
  • CAP theorem: under a network partition you must pick consistency or availability; you cannot have both
  • Backpressure: when a downstream consumer cannot keep up, push the slowdown back to the producer instead of dropping or buffering forever
  • Idempotency: an operation is idempotent if running it twice produces the same result as running it once; the property that makes safe retry possible
  • Retry: try the same operation again after a failure; only safe when the operation is idempotent
  • Reliability: the probability the system does the right thing under stress

Unit 5 Queues, Retries, the Distributed Cipher Lab

What we covered

SQS visibility timeout, why the same message can be delivered twice, how to design a worker so that double delivery does not produce double side effects. The Distributed Cipher lab as a place where all of this had to be real, not theoretical.

Flashcards
  • What is a queue: a buffer between a producer and a consumer; producers add messages, consumers pull them
  • Why use a queue: decoupling (the producer does not have to wait for the consumer), smoothing (the queue absorbs traffic spikes), durability (the message survives if the consumer crashes mid-work)
  • Producer: the thing that puts work into the queue
  • Consumer / worker: the thing that takes work out of the queue and processes it
  • Message: the unit of work; usually JSON, with a body and optional attributes
  • Synchronous vs asynchronous: caller waits for the result vs caller fires and forgets; the queue is what makes it asynchronous
  • What is SQS: Simple Queue Service, AWS's managed message queue; you create a queue and send and receive messages over an HTTPS API
  • Standard vs FIFO queue: high throughput, possible duplicates and reorder, vs ordered, deduplicated, lower throughput
  • Visibility timeout: after a worker receives a message, SQS hides it from other workers for a configurable window; if the worker does not delete it in that window, it reappears
  • At-least-once delivery: the same message can arrive twice; you must design for it
  • Idempotent worker: double delivery produces no extra side effects (same result, no duplicate writes downstream)
  • Dead letter queue (DLQ): catches messages that fail repeatedly so the main queue does not get blocked behind a poison message
  • Redrive policy: the rule that says "after N failed receives, send to the DLQ"
  • Batching: up to 10 messages per send or receive call; amortizes the per-call cost
  • Long polling: the receive call waits up to 20 seconds for a message instead of returning empty immediately; cheaper and faster than constant short polls
  • "Exactly once" delivery is a fairy tale you build out of at-least-once plus an idempotent consumer
  • Distributed Cipher lab: where all of this had to be real, not theoretical

Unit 6 pandas to Polars

What we covered

Why pandas runs out of memory on a single laptop with multi-gigabyte files. Polars's lazy API, predicate pushdown, projection pushdown, and the Arrow columnar format underneath both Polars and Spark. The single-machine sweet spot before you have to reach for a cluster.

Flashcards
  • What is a dataframe: a table-like data structure with named columns and typed values; the standard tool for tabular analysis in Python
  • What is pandas: the original Python dataframe library; in-memory, eager, single-threaded; the default for the last fifteen years
  • What is Polars: a newer dataframe library written in Rust; columnar, lazy, multi-threaded; same idea as pandas, faster engine
  • pandas at a glance: row-oriented in memory, runs out of room around 1 to 2 GB on a typical laptop
  • Polars at a glance: column-oriented in memory, scales to roughly the size of your RAM
  • Eager evaluation: each operation runs immediately, in order, exactly as written; pandas does this
  • Lazy evaluation: operations build a plan; the engine sees the whole plan before running it and can optimize; Polars and Spark do this
  • What is the GIL (Global Interpreter Lock): Python's rule that only one thread runs Python bytecode at a time; the reason pure-Python loops in pandas cannot use more than one core
  • Vectorization: replace Python loops with C loops over arrays; the trick that makes NumPy, pandas, and Polars fast
  • Row-based vs columnar storage: rows together on disk and in memory vs columns together; columnar wins for analytics because you usually want a few columns over many rows
  • What is predicate pushdown: the engine pushes WHERE filters down into the file scan, so unmatching rows are never read
  • Effect of predicate pushdown: dramatically less I/O on large Parquet files; almost always a win, only "bad" if your filter is so loose that the engine still scans almost everything
  • Where it lives: in the query engine (Polars, Spark, DuckDB) plus the file format (Parquet stores per-row-group min and max so the engine can skip whole chunks); pandas does not do this
  • What is projection pushdown: the engine pushes column selection down into the file scan, so unwanted columns are never read
  • Effect of projection pushdown: a query that uses 3 columns out of 50 reads roughly 6% of the bytes; combined with predicate pushdown a query can read less than 1% of a Parquet file
  • What is Apache Arrow: a shared columnar in-memory format; lets Polars, Spark, DuckDB, and pandas pass data around with no copy or serialization
  • What is Parquet: a columnar file format on disk; compressed, schema-aware; the standard storage format under modern data tools
  • What is DuckDB: a single-machine analytics engine, SQL on top of Parquet and Arrow; the "SQLite for analytics"
  • What is Dask: a pandas-style API spread across many cores or many machines; the way to scale pandas without rewriting

Unit 7 Spark, the Lakehouse, and Unity Catalog

What we covered

Spark transformations are lazy and only execute when an action is called (the DAG, the Catalyst optimizer). Unity Catalog as "DNS for data" with its three-level namespace (catalog.schema.table). Why a columnar query on Parquet is so much faster than the same query on CSV. The Lakehouse pattern as the answer to "data swamps".

Flashcards
  • What is Spark (Apache Spark): an open-source distributed compute engine for large datasets; you write transformations in Python, SQL, or Scala and Spark runs them in parallel across a cluster
  • What is Databricks: the commercial cloud platform built around Spark; adds notebooks, job scheduling, Unity Catalog, Delta Lake, and a managed runtime
  • What is a cluster: a set of machines working together as one logical compute resource; in Spark, one driver and many executors
  • What is a DAG (Directed Acyclic Graph): a graph of steps where each step has dependencies (edges) and there are no cycles; Spark represents your query as a DAG of transformations before it runs anything
  • What is lazy evaluation: the engine records what you asked for but does not run it; only an action (like .show(), .count(), or .write()) forces execution of the whole DAG at once
  • Transformation vs action: transformations (filter, select, join) are lazy and just add to the DAG; actions trigger the engine to actually compute
  • Why lazy is good: the engine sees the full plan and can optimize across stages (push filters down, prune columns, reorder joins); eager engines cannot
  • What is the Catalyst optimizer: Spark's query planner; it takes your DAG and rewrites it (predicate pushdown, projection pushdown, join reordering) before generating the physical plan
  • Driver vs executor: the driver is the single coordinator that holds your code and the plan; executors are the worker processes that actually run the tasks on data partitions
  • What is a partition (in Spark): a chunk of the dataset assigned to one executor; parallelism in Spark is one task per partition
  • What is a shuffle: the expensive operation where data has to move between executors over the network (e.g., a wide join or a group-by); the thing you try to minimize
  • Broadcast join: a small table is sent in full to every executor so the join can happen locally with no shuffle; only safe when the small side is actually small
  • What is the Lakehouse: an architecture pattern that puts warehouse-grade features (ACID transactions, schema, governance) on top of cheap object storage; the answer to "data swamps"
  • What is Parquet: a columnar file format; compressed, schema-aware, splits data into row groups with min and max statistics so scanners can skip whole chunks
  • Parquet vs CSV: Parquet is columnar, compressed, typed; CSV is row-based plain text with no types; Parquet scans can be 10 to 100 times faster
  • What is Delta Lake: an open table format that adds a transaction log on top of Parquet files; gives you ACID transactions, schema enforcement, and updates and deletes on object storage
  • What is ACID: Atomicity, Consistency, Isolation, Durability; the four guarantees that make a write either fully succeed or fully not happen, even with concurrent writers
  • Time travel: VERSION AS OF or TIMESTAMP AS OF on a Delta table reads it as it was at that commit; useful for debugging and reproducibility
  • What is Unity Catalog: Databricks' metadata and access-control layer; one place that knows what tables exist, what their schemas are, and who can read or write them
  • Three-level namespace: catalog.schema.table; the catalog is the top container, schema (a.k.a. database) groups related tables, table is the actual data
  • "DNS for data": the metaphor for Unity Catalog; the same name resolves to the same physical table no matter who is asking, with permissions enforced centrally

Unit 8 Medallion Architecture: Bronze, Silver, Gold

What we covered

Bronze as raw, append-only, all-strings, untouched. Silver as cleaned, typed, deduplicated, modeled. Gold as business-ready aggregates and joins. The custodian-report worked example. Why separating Silver from Gold pays off as soon as more than one team needs the cleaned data.

Flashcards
  • What is a data architecture: the layered plan for how raw data turns into useful tables; defines what lives where, who owns it, and what shape it has at each step
  • What is the Medallion Architecture: a three-tier convention (Bronze, Silver, Gold) for organizing tables in a Lakehouse, popularized by Databricks; one of several layouts you might use
  • Bronze (raw): data as it landed; append-only, all strings, no transformations, no parsing; the system of record for "what arrived"
  • Silver (cleaned): typed, deduplicated, joined to reference data, conformed to a model; one canonical version of each entity
  • Gold (business-ready): aggregates, joins, and views shaped for a specific consumer (a dashboard, a report, a model); read-optimized
  • Why split Silver from Gold: many Gold tables (one per audience) can share the same Silver source; you avoid recomputing cleaning logic in each report
  • Reprocess from Bronze if Silver logic has a bug; you do not need to re-ingest from the source system because Bronze still has the raw bytes
  • Idempotent Bronze ingest: the same source file landed twice produces no duplicates downstream; usually achieved by including the file name and offset in the row key
  • What is SCD (Slowly Changing Dimension): the pattern for handling attributes that change over time (a customer's address, a product's price)
  • SCD Type 1 vs Type 2: overwrite the old value (no history) vs keep history with valid_from and valid_to columns
  • What is a star schema: a fact table in the middle (one row per event or transaction) with dimension tables hanging off it (one row per entity); the classic warehouse layout
  • Fact vs dimension: facts are measurements (sales amount, click count), dimensions are descriptors (customer, product, date)
  • Custodian-report worked example: the in-class case where the same Silver layer fed three different Gold reports without recomputing the cleaning logic

Unit 9 Declarative Pipelines and dbt

What we covered

The imperative way: notebooks chained by Workflows, you wire the dependency graph by hand. The declarative way: @dlt.table() in Lakeflow Declarative Pipelines (DLT), where you describe the contract and the platform handles ordering, retries, and quality expectations. dbt as the SQL-first cousin: every model is a SELECT, ref() and source() build the DAG, the warehouse does the work.

Flashcards
  • What is a data pipeline: a chain of steps that moves and transforms data from a source to a usable table; runs on a schedule or in response to new data
  • Imperative style: you write the steps in order and wire the dependencies by hand; you say how to do each thing; notebooks chained by a Workflow are imperative
  • Declarative style: you describe what the output table should be; the platform figures out the order, retries, and incrementality; @dlt.table() is declarative
  • What is a Workflow (Databricks): a job scheduler in Databricks; lets you chain notebooks or scripts in a DAG with retry policies and a schedule
  • What is DLT (Lakeflow Declarative Pipelines, formerly Delta Live Tables): a Databricks framework where you decorate a function with @dlt.table() and the platform manages the DAG, scheduling, and quality enforcement for you
  • DLT expectations: data-quality rules attached to a table; rows that fail can be dropped, quarantined, or fail the run
  • Streaming table: append-only, processed incrementally as new data arrives
  • Materialized view: the result of a query, recomputed on a schedule or when its inputs change
  • What is dbt (data build tool): a SQL-first transformation framework; you write SELECT statements, dbt builds the DAG, runs them in the right order in your warehouse, and tests the outputs
  • dbt model: a single .sql file that contains one SELECT; the file name becomes the table name
  • ref('other_model'): dbt's way of saying "this model depends on that one"; this is how dbt builds the DAG, and it lets dbt swap in dev vs prod schemas automatically
  • source('schema','table'): a reference to a raw upstream table that dbt did not create; lets you test and document the inputs to your pipeline
  • Materializations in dbt: view (cheap, recomputed on every read), table (full rebuild each run), incremental (only new rows), ephemeral (compiled inline as a CTE, no object created)
  • Incremental model needs a watermark column (typically updated_at or an event timestamp) so dbt knows which rows are new since the last run
  • What is Jinja: a Python templating language; lets you put {{ variables }}, {% for %} loops, and macros inside what would otherwise be plain SQL; dbt is essentially "SQL with Jinja on top"
  • What is a macro (in dbt): a reusable Jinja function that generates SQL; lets you stop copy-pasting the same logic across models
  • dbt snapshot: the built-in helper for SCD Type 2 history tracking on a source table
  • dbt tests: built-in checks like unique, not_null, accepted_values, plus custom SQL tests; run as part of the build
  • What is Airflow: the older general-purpose Python-based pipeline orchestrator; Workflows is the Databricks-native equivalent for the same job

Unit 10 MLOps

What we covered

The dev / staging / production framework. MLflow as the experiment tracker (params, metrics, artifacts, every run logged). The Model Registry as named, versioned models with stage transitions. The "imagine if all the engineers in the world suddenly lost GitHub" framing for why this matters in data science.

Flashcards
  • What is MLOps: the practices and tools for taking a model from a notebook on someone's laptop to a versioned, monitored thing running in production; "DevOps for machine learning"
  • Why it exists: models trained once on a static dataset behave differently in production where data shifts, code changes, and dependencies move; you need versioning, lineage, and rollback or you cannot debug
  • What is an experiment (in ML terms): one training run with a specific dataset, code, and set of hyperparameters; you typically run hundreds while developing
  • Three environment stages: dev (your sandbox), staging (a production-like rehearsal), production (real traffic, real users)
  • Why three stages: dev catches bugs, staging catches integration problems and performance regressions, production is what the business depends on
  • What is MLflow: an open-source tool for tracking ML experiments and managing trained models; the de facto standard, bundled into Databricks
  • MLflow tracking: for each run, log parameters (the hyperparameters), metrics (accuracy, loss), and artifacts (the trained model file, plots, the dataset hash)
  • What is the Model Registry: the part of MLflow that stores named, versioned models; lets you say "model X version 7 is now in production" without redeploying code
  • Stage transitions: a model version moves from None to Staging to Production to Archived; the transition is the audit trail
  • Reproducibility: log enough (code commit, data version, library versions, random seed) that you can rerun the experiment six months later and get the same answer
  • What is a feature store: a shared, versioned table of model inputs (features); the same feature definition is used for both training and live serving so they stay in sync
  • Training-serving skew: the bug where the same feature is computed one way during training (batch SQL) and a different way during serving (live Python); the model sees different distributions and quietly degrades
  • Model drift: the model's predictions get worse over time even though the input distribution looks the same (concept change)
  • Data drift: the inputs themselves shift; the model is doing the same thing, but the world changed under it
  • Canary deployment: route a small slice of traffic (say 5%) to the new model version, watch its metrics, and ramp up only if it behaves
  • Champion-challenger (a.k.a. A/B for models): run two model versions side by side on real traffic and compare
  • What is pickle: Python's built-in object serialization; convenient and the default way most ML libraries save models
  • Why pickle is a security risk: unpickling a file executes arbitrary Python code; a malicious model file can run anything when loaded; never load a pickle from an untrusted source
  • "GitHub for models": the framing for the Model Registry; versioning, lineage, rollback, code review
  • Lineage: the recorded chain "this model came from that training run, which used that data version, which came from that pipeline"; what makes audits and rollbacks possible

Unit 11 Kafka and Event Streaming

What we covered

Topics, partitions, offsets. The producer key as the routing input (hash(key) mod num_partitions) and what that gives you (per-key ordering) and what it costs you (a hot key is a hot partition). Consumer groups, rebalancing, the greenhouse-sensor lab.

Flashcards
  • What is event streaming: a data pattern where systems publish small immutable records (events) as they happen, and other systems subscribe and react in near real time
  • Event vs message: an event is a fact that happened ("user clicked button X at time T"); a message is a command or request to be processed; Kafka carries events
  • What is Kafka (Apache Kafka): a distributed, append-only, log-based event streaming platform; producers write events, brokers store them, consumers read them at their own pace
  • Why Kafka and not SQS: Kafka keeps the log around for days or weeks so many independent consumers can replay; SQS deletes a message once it is acknowledged
  • What is a broker: one Kafka server in the cluster; brokers store partitions and serve reads and writes
  • What is a topic: a named append-only log; the unit of organization (e.g., orders, sensor-readings)
  • What is a partition: a topic is split into N partitions; each partition is its own ordered log on disk; partitions are how Kafka scales horizontally
  • What is an offset: the position of a record inside a partition; monotonically increasing, never reused
  • The three coordinates: topic, partition, offset uniquely identify any record in Kafka
  • What is a producer: the client that publishes events to a topic
  • Producer key: an optional value attached to each event; if set, the producer routes via hash(key) mod num_partitions to pick the partition
  • Same key, same partition: all events with the same key land in the same partition, in the order they were sent; this is how you get per-key ordering across a distributed log
  • Hot key: one key value gets disproportionate traffic; that single partition becomes a bottleneck while the others sit idle; the cost of ordering
  • What is a consumer: the client that reads events from a topic; tracks its own offset
  • Consumer group: a set of consumers cooperating to read a topic; each partition is assigned to exactly one member of the group at a time, so the work is split
  • Rebalancing: when a consumer joins or leaves the group, Kafka reassigns partitions; brief pause in processing while it happens
  • Replication factor: how many brokers hold a copy of each partition; 3 is typical (one leader, two followers); survives broker failures
  • What is Avro / Protobuf: two compact binary serialization formats with explicit schemas; the standard alternatives to JSON for high-volume topics
  • What is the Schema Registry: a separate service that stores the Avro or Protobuf schema for each topic and refuses incompatible producer changes; contract enforcement
  • Log compaction vs time retention: compaction keeps only the latest value per key (good for "current state" topics); retention deletes records older than N days regardless of key
  • What is Kafka Connect: a framework of pre-built connectors that move data between Kafka and external systems (databases, S3, search) without writing code
  • What is Kafka Streams / ksqlDB: stream-processing libraries that let you run continuous SQL or Java transformations against Kafka topics
  • Greenhouse-sensor lab: the in-class place where keyed routing, partitions, and consumer groups stopped being theory

Unit 12 Specialized Compute (Runpod, GPUs)

What we covered

Runpod serverless workers, the GPU vs CPU split, queue-based workers vs load-balanced HTTP endpoints. Compute as something you rent for ninety seconds at a time. The economics of "I do not own any of this hardware and I never will."

Flashcards
  • What is a CPU: the general-purpose processor; a few powerful cores optimized for branching code and a wide range of instructions
  • What is a GPU: the graphics processor; thousands of small cores optimized for doing the same operation on a lot of numbers at once (matrix math)
  • Why GPUs for ML: training and inference are mostly large matrix multiplies; that is exactly what GPU hardware is built for
  • What is specialized compute: hardware purpose-built for one class of workload (GPU for matrix math, TPU for tensor ops, FPGA for custom logic) instead of a general CPU
  • What is Runpod: a cloud provider that rents GPUs by the second, including a serverless option where you pay only when a request is actually running
  • Serverless GPU model: the platform spins up a GPU container on demand, runs your code (typically tens of seconds for inference), and tears it down; you pay per request, not per hour
  • Queue worker (async): client submits a job, gets a job ID, polls or gets a webhook when it is done; good for long jobs and bursty load
  • HTTP endpoint (sync): client sends a request and blocks waiting for the response; good for low-latency interactive use
  • Cold start on GPU is real: the model weights (often gigabytes) have to load into GPU memory before the first request can run; can dwarf the actual inference time
  • What is model inference: running a trained model forward on new inputs to get predictions; the production-time half of ML
  • What is quantization: storing model weights in fewer bits (int8, int4) instead of float32; smaller memory footprint, faster compute, small accuracy hit
  • Batching: hold a few requests for a few milliseconds, run them through the GPU together; amortizes the per-call overhead and dramatically increases throughput
  • What is vLLM: a popular open-source inference server for large language models; handles continuous batching and KV-cache reuse so you do not have to
  • What is TGI (Text Generation Inference): Hugging Face's inference server; same job as vLLM, different implementation
  • What is the KV cache: the per-request attention state in a transformer; reusing it across tokens is what makes generation fast
  • Container image size matters: bigger image means slower pull from the registry on cold start; keep base images lean
  • VRAM (GPU memory) is the binding constraint: a model that does not fit in VRAM either does not run or has to spill to slow host memory
  • You never own the hardware: the entire economic story of cloud compute; you rent capacity at the granularity (seconds, minutes, hours) that matches your workload
