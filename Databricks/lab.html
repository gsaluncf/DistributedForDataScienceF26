Stage 2 Transform — Build a Customer 360 View

Your goal: create a single denormalized table that combines customer info, their order history, and geographic data. This is the classic "Customer 360" pattern — one row per customer with all their metrics.

2.1 The Customer 360 Table

Task 2.1 — Build the Pipeline

Write a CREATE OR REPLACE TABLE statement that joins all four tables and computes per-customer aggregates. Your table should be called workspace.default.customer_360.

Required columns:

  • customer_id — the customer key
  • customer_name
  • segment — market segment (BUILDING, AUTOMOBILE, etc.)
  • nation — country name
  • region — continent name
  • total_orders — count of all orders
  • total_revenue — sum of order totals (rounded to 2 decimal places)
  • avg_order_value — average order total (rounded to 2 decimal places)
  • first_order_date — earliest order date
  • last_order_date — latest order date
  • customer_tenure_days — days between first and last order

Hint: You need a 3-way JOIN (customer → orders, customer → nation, nation → region) plus GROUP BY. Use DATEDIFF for tenure.

Check the Query Profile

After running your CTAS, click "See performance". Look for: the Exchange (shuffle) nodes — how many are there? This query joins 4 tables, so Spark must redistribute data multiple times. Count the shuffles and note how many rows each one moved. You will reference this in your reflection.

2.2 Verify Your Table

Task 2.2 — Sanity Checks

Run these queries against your new table. If any fail, your transform has a bug.

-- Should be exactly 750,000 (one row per customer)
SELECT COUNT(*) AS total_customers FROM workspace.default.customer_360;

-- Should be 5 segments
SELECT segment, COUNT(*) AS n FROM workspace.default.customer_360
GROUP BY segment ORDER BY n DESC;

-- Should be 5 regions
SELECT region, COUNT(*) AS n FROM workspace.default.customer_360
GROUP BY region ORDER BY n DESC;

-- Spot-check: no NULLs in critical columns
SELECT
  SUM(CASE WHEN customer_id IS NULL THEN 1 ELSE 0 END) AS null_id,
  SUM(CASE WHEN total_orders IS NULL THEN 1 ELSE 0 END) AS null_orders,
  SUM(CASE WHEN region IS NULL THEN 1 ELSE 0 END) AS null_region
FROM workspace.default.customer_360;

Stage 3 Analyze — Build Analytics on Top

Now that you have a clean, denormalized table, build analytics views on top of it. This is the "reporting layer" — the queries that a dashboard or business analyst would run.

3.1 Segment Performance Summary

Task 3.1 — Segment Report

Write a query that produces a segment-level summary:

  • Number of customers per segment
  • Total revenue per segment
  • Average order value per segment
  • Median customer tenure (days) per segment — use PERCENTILE_APPROX(customer_tenure_days, 0.5)

Order by total revenue descending. Materialize the result as workspace.default.segment_report.

3.2 Top Customers by Region (Window Function)

Task 3.2 — Top 5 per Region

Using a window function, find the top 5 customers by total revenue within each region.

Required output columns: region, customer_name, segment, nation, total_revenue, rank_in_region

Hint: Use RANK() OVER (PARTITION BY region ORDER BY total_revenue DESC), then filter for rank ≤ 5.

Materialize as workspace.default.top_customers_by_region.

3.3 Revenue by Year and Region (Trend Analysis)

Task 3.3 — Yearly Trends

This one requires going back to the raw samples.tpch.orders table (since your customer_360 already aggregated order dates). Write a query that shows:

  • Year (from o_orderdate)
  • Region
  • Total revenue for that year+region
  • Year-over-year change as a percentage — use LAG() window function

Hint: Join orders → customer → nation → region, group by year + region, then use LAG(total_revenue) OVER (PARTITION BY region ORDER BY year) to get the previous year's revenue.

Materialize as workspace.default.yearly_trends.

Stage 4 Optimize — Partitioning Experiment

Your customer_360 table was written as a single Delta table. Now create a partitioned copy and compare query performance.

4.1 Create a Partitioned Copy

Task 4.1 — Partition by Region
CREATE OR REPLACE TABLE workspace.default.customer_360_partitioned
PARTITIONED BY (region)
AS SELECT * FROM workspace.default.customer_360;

4.2 Compare Query Profiles

Task 4.2 — Side-by-Side Comparison

Run the exact same query on both tables:

-- Query A: Non-partitioned
SELECT segment, COUNT(*), ROUND(AVG(total_revenue), 2) AS avg_rev
FROM workspace.default.customer_360
WHERE region = 'EUROPE'
GROUP BY segment ORDER BY avg_rev DESC;

-- Query B: Partitioned
SELECT segment, COUNT(*), ROUND(AVG(total_revenue), 2) AS avg_rev
FROM workspace.default.customer_360_partitioned
WHERE region = 'EUROPE'
GROUP BY segment ORDER BY avg_rev DESC;

Open the Query Profile for both. Look at the Scan node:

  • How many bytes did Query A scan?
  • How many bytes did Query B scan?
  • What is the ratio?

The partitioned table should scan significantly less data because it prunes the partitions it does not need. This is partition pruning — one of the most important optimization techniques in distributed query engines.

Why Partitioning Matters

In a distributed system, data is spread across workers. When you query a partitioned table with a filter on the partition column (WHERE region = 'EUROPE'), Spark can skip entire files on disk — workers that hold other regions never even wake up. This reduces I/O, network traffic, and wall-clock time. It is the same principle as sharding in databases and consistent hashing in distributed caches.

Stage 5 Reflect — Distributed Systems Connections

Create a final markdown cell in your notebook titled "Reflection" and answer the following questions. These connect the lab to broader distributed systems concepts from the course.

Task 5.1 — Reflection Questions
  1. Shuffles and coordination cost. In Stage 2, your Customer 360 query triggered multiple shuffle (Exchange) operations. Each shuffle redistributes data across workers over the network. Why are shuffles the most expensive part of distributed query execution? What is the coordination overhead involved?
  2. Partition pruning vs. full scan. In Stage 4, you compared partitioned and non-partitioned tables. How is partition pruning similar to the concept of sharding in distributed databases? What happens if you pick a bad partition key (e.g., a column with millions of unique values)?
  3. Denormalization tradeoffs. The Customer 360 table is denormalized — region and nation names are stored in every row instead of being joined at query time. In a distributed system, why is denormalization often preferred even though it duplicates data? What problems does it create?
  4. Abstraction layers. You wrote this entire pipeline in SQL. Spark compiled each query into a distributed execution plan (tasks, stages, shuffles) that ran across multiple workers — and you never configured any of that. What are the advantages of this abstraction? What are the risks of not understanding what happens underneath?
  5. Connecting to the semester. We have studied Lambda/SQS (event-driven), P2P gossip (decentralized), and now Spark (coordinator-worker). Compare how these three systems handle data distribution and fault tolerance. Which is most similar to what you did today? Which is most different?
