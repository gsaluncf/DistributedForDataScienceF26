This week, we move to the data side of distributed computing. Instead of building infrastructure, you will use Apache Spark — a distributed query engine — to process data at a scale that would bring a single-server database to its knees.

The Core Idea

Spark splits your data across many machines and runs your queries in parallel. You write code that looks like it operates on one table — but it actually runs on a cluster of workers. This is the same principle as your Lambda functions processing SQS messages in parallel, but applied to analytics instead of event processing.

Part 1 Apache Spark — Distributed Query Engine

1.1 From Single Server to Distributed Compute

In a traditional database (PostgreSQL, MySQL), everything happens on one machine: the data sits on one disk, the query runs on one CPU, and the result goes to one client. This works beautifully for millions of rows.

But consider a dataset with 30 million rows and 16 columns (the lineitem table you will use this week). A single-server scan takes seconds. Now imagine 30 billion rows — 1,000x larger. A single machine would take hours.

Spark's Approach

Spark splits the table into partitions — chunks of data assigned to different workers. Each worker processes its chunk independently. A query that takes 60 minutes on one machine might take 1 minute across 60 workers. This is horizontal scaling — the same principle you used when multiple Lambda functions processed SQS batches in parallel.

Concept Traditional Database Apache Spark
Data location One server's disk Distributed across workers
Query execution Single thread (mostly) Parallel across cluster
Data format Proprietary (MySQL pages, Postgres heap) Open formats (Parquet, Delta, Iceberg)
Scaling Buy a bigger server (vertical) Add more workers (horizontal)
API SQL only SQL + Python + Java + Scala

1.2 DataFrames — Distributed Tables

A Spark DataFrame looks and feels like a pandas DataFrame — you can .filter(), .groupBy(), .join() — but under the hood, the data lives across multiple machines. Spark translates your operations into a distributed execution plan.

# Load a table — data is distributed across workers automatically
df = spark.read.table("samples.tpch.customer")

# Filter + select — these are TRANSFORMATIONS (lazy, no execution yet)
building = df.filter(df.c_mktsegment == "BUILDING")
selected = building.select("c_custkey", "c_name", "c_acctbal")

# Count — this is an ACTION (triggers distributed execution)
count = selected.count()  # → 150,297

1.3 Lazy Evaluation & the DAG

The most important concept in Spark: transformations are lazy. When you write df.filter(...).select(...).groupBy(...), Spark does nothing. It records your intent in a plan called a DAG (Directed Acyclic Graph). Only when you call an action (.count(), .show(), display()) does Spark compile the DAG, optimize it, and execute it.

Transformations are recorded lazily. Only an Action triggers the distributed execution plan.

Why lazy? Because the Catalyst optimizer can see your entire chain of transformations and rearrange them for efficiency. For example, if you filter after a join, Spark moves the filter before the join (predicate pushdown) — so fewer rows need to be shuffled.

2.1 Row-Based vs Columnar Storage

This is the most fundamental concept in data engineering. Consider a table with 3 columns and millions of rows:

Row-based storage reads everything. Columnar storage reads only the columns your query needs (column pruning).

2.2 The Five Formats

Format Storage Schema Compression ACID Best For
CSV Row-based None (guessed) None No Small data, human-readable exchange
Parquet Columnar Embedded (exact) Built-in (Snappy) No Analytics, data lakes
Avro Row-based Embedded (exact) Built-in No Streaming, Kafka, event logs
Delta Lake Columnar (Parquet) Enforced Built-in Yes Production data systems
Iceberg Columnar (Parquet) Enforced Built-in Yes Multi-engine environments

2.3 Delta Lake — Parquet + Transactions

Delta Lake is the most important format for production data systems. It takes standard Parquet files and adds a transaction log that enables:

  • ACID transactions — writes are atomic. A crash mid-write cannot corrupt your table.
  • Time travel — query any previous version with VERSION AS OF n
  • Schema enforcement — rejects writes with wrong data types
  • UPDATE / DELETE — modify individual rows without rewriting the whole table
# Delta Lake on disk:
my_table/
├── _delta_log/               ← Transaction log (JSON files)
│   ├── 000000.json           ← Version 0: "created table, added part-001.parquet"
│   ├── 000001.json           ← Version 1: "UPDATE 500 rows, added part-003.parquet"
│   └── 000002.json           ← Version 2: "DELETE WHERE status = 'cancelled'"
├── part-001.snappy.parquet   ← Standard Parquet data files
├── part-002.snappy.parquet
└── part-003.snappy.parquet

# Time travel — read old versions:
spark.read.format("delta").option("versionAsOf", 0).table("my_table")

Part 3 Unity Catalog — DNS for Data

3.1 The Service Discovery Problem

In every distributed system, services need to find things. Recall:

  • P2P project: nodes discovered peers via a gossip protocol
  • AWS project: Lambda found queues via SQS URLs, tables via DynamoDB names
  • DNS: browsers find servers by resolving google.com → 142.250.80.46
  • Kubernetes: pods find services via the cluster's service registry

In a data lakehouse, the problem is: hundreds of notebooks need to find thousands of tables, with access controls, versioning, and lineage tracking.

Unity Catalog = DNS for Data

Unity Catalog is a centralized registry for all data assets. Every table, view, file, and ML model gets a three-level address: catalog.schema.table. Any notebook, job, or SQL query can find any asset using this address — like DNS for data.

4.2 The Execution Plan — What Spark Actually Does

Every Spark query (SQL or DataFrame) goes through four planning stages:

  1. Parsed Logical Plan — your raw SQL/code converted to an abstract syntax tree
  2. Analyzed Logical Plan — column and table names resolved against Unity Catalog
  3. Optimized Logical Plan — Catalyst optimizer applies rules (predicate pushdown, column pruning, join reordering)
  4. Physical Plan — actual execution strategy: which join algorithm, where to shuffle, how to scan

View with df.explain(True) or click "See performance" after any cell to open the Query Profile — a visual DAG showing exactly how Spark executed your query.

Part 7 A Brief History of Databricks

In 2009, the AMPLab (Algorithms, Machines, and People Lab) at UC Berkeley set out to build tools for large-scale data analysis. PhD student Matei Zaharia created Apache Spark in 2010 as a faster alternative to Hadoop MapReduce — keeping data in memory instead of writing intermediates to disk.

In 2013, Zaharia and other Spark creators founded Databricks to commercialize the technology. The platform evolved significantly:

Year Milestone Why It Matters
2009 AMPLab at UC Berkeley Research lab where Spark was born
2010 Apache Spark created In-memory processing — 100x faster than MapReduce for iterative workloads
2013 Databricks founded Managed cloud platform around Spark
2017 Delta Lake introduced ACID transactions on data lakes — solved reliability problems
2020 Lakehouse architecture Unified data warehouses and data lakes into one platform
2023+ DBSQL / AI era SQL-first workflows, serverless compute, AI agents, Unity Catalog

7.1 Why Databricks?

In a distributed systems course, we care about Databricks because it is the most widely-adopted commercial implementation of distributed data processing. It shows how the industry moved from "you manage the distributed system" (install Hadoop, configure YARN, tune Spark) to "the platform manages the distributed system" (write SQL, the platform handles parallelism and fault tolerance).

The key idea is convergence: instead of separate systems for your data warehouse, data lake, ML pipelines, and ETL jobs, Databricks puts everything on one platform with one governance model. This is the Lakehouse architecture.

7.2 Why PySpark Still Matters

Even though the industry is shifting toward DBSQL, PySpark remains essential:

  • It reveals the distributed system — PySpark forces you to think about partitions, shuffles, and transformations vs actions. DBSQL hides this.
  • Legacy codebases are everywhere — hundreds of thousands of PySpark jobs run in production. You will encounter code that needs maintaining or migrating.
  • Complex transformations need it — custom UDFs, streaming pipelines, and ML feature engineering often require PySpark or Scala.
  • Interview expectations — data engineering interviews still test PySpark knowledge.
Bottom Line

DBSQL is how you will probably work day-to-day. PySpark teaches you what is actually happening underneath. Understanding both levels — the abstraction and the mechanism — is what makes you effective.

Further Reading

Apache Iceberg — The Other Table Format

We use Delta Lake in this course because it is Databricks' native format. But Apache Iceberg (created at Netflix) solves the same problem: ACID transactions, schema evolution, and time travel on top of data lake files. Its killer feature is engine independence — one Iceberg table can be read by Spark, Trino, Flink, Snowflake, and BigQuery simultaneously. Knowing both formats shows interviewers you understand the landscape, not just one vendor.
