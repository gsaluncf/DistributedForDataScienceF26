What Is Apache Iceberg?

A brief overview of the open table format that competes with Delta Lake.

The Problem Iceberg Solves

Parquet files are great for columnar storage, but they are just files — no transactions, no schema evolution, no way to safely update a row without rewriting the whole file. Table formats add a metadata layer on top of files to make them behave like proper database tables.

Two projects emerged to solve this: Delta Lake (Databricks, 2017) and Apache Iceberg (Netflix, 2018). Both add ACID transactions, time travel, and schema enforcement to data lakes. They solve the same problem differently.

What Is Iceberg?

Apache Iceberg is an open table format for large analytic datasets. Created at Netflix to manage petabyte-scale tables, it was donated to the Apache Software Foundation in 2018. Key idea: instead of using a transaction log (Delta Lake's approach), Iceberg uses a tree of metadata files that track which data files make up each version of a table.

  • Manifest files — list individual data files and their column-level stats
  • Manifest lists — group manifests into a snapshot
  • Snapshots — immutable pointers to a specific version of the table
  • Catalog — tracks the current snapshot (the "pointer to the pointer")

This layered metadata design gives Iceberg extremely efficient partition pruning and file skipping — the query engine can skip files without opening them, just by reading the manifest stats.

Iceberg vs Delta Lake

Feature Delta Lake Apache Iceberg
Created by Databricks (2017) Netflix (2018)
Metadata approach Transaction log (_delta_log/) Manifest tree (snapshots → manifest lists → manifests)
ACID transactions Yes Yes
Time travel Yes Yes
Schema evolution Add/rename columns Add/rename/reorder/drop columns (more flexible)
Partition evolution Must rewrite table Change partitioning without rewriting data
Hidden partitioning No — users must know partition columns Yes — Iceberg auto-derives partitions from expressions
Engine support Spark, Databricks, some others Spark, Trino, Flink, Dremio, Snowflake, BigQuery, Databricks
Governance Tightly integrated with Databricks / Unity Catalog Engine-agnostic — works across platforms

Why Iceberg Matters

Iceberg's big selling point is engine independence. A single Iceberg table can be read by Spark, Trino, Flink, Snowflake, and BigQuery simultaneously — no vendor lock-in. This is why many organizations that run multi-engine environments choose Iceberg.

Delta Lake's strength is deep integration with Databricks — optimizations like Z-ordering, Liquid Clustering, and the Unity Catalog governance layer work best (or only) with Delta. If you are all-in on Databricks, Delta is the natural choice.

The Industry Trend

The lines are blurring. Databricks added Iceberg read support (UniForm), and Snowflake adopted Iceberg as a native format. The industry is converging on the idea of open table formats — the debate is less about Delta vs Iceberg and more about having ACID transactions and governance on top of open file storage.

Hidden Partitioning — Iceberg's Signature Feature

In Delta Lake (and traditional Hive tables), partitioning requires users to know the exact partition column and write queries that filter on it. Get it wrong and you get a full scan.

Iceberg introduces hidden partitioning: you define partition transforms (like "month of order date") and Iceberg automatically applies them. Users write normal queries and the engine handles partition pruning transparently.

-- Delta Lake: user must know to filter on the partition column
SELECT * FROM orders WHERE order_month = '2024-01'

-- Iceberg: user writes a normal query, Iceberg prunes automatically
SELECT * FROM orders WHERE order_date BETWEEN '2024-01-01' AND '2024-01-31'
-- Iceberg knows order_date is partitioned by month and skips irrelevant files

Where You Will See Iceberg

  • Multi-cloud / multi-engine shops — organizations using Spark + Trino + Snowflake
  • AWS — Athena and EMR have strong Iceberg support
  • Snowflake — adopted Iceberg as a native managed table format
  • Databricks — supports reading Iceberg via UniForm; Delta remains the default
For This Course

We use Delta Lake because we are on Databricks Free Edition and Delta is the native default. But knowing that Iceberg exists — and understanding that the core idea (ACID + metadata on top of files) is the same — puts you ahead. In interviews, being able to explain the Delta vs Iceberg tradeoff shows you understand the landscape, not just one vendor's tool.
