Data Pipelines & the Medallion Architecture

How data gets into the Lakehouse and flows through layers of increasing quality. From a raw custodian Excel file to a production-ready portfolio valuation, using Unity Catalog, Delta Lake, and Databricks Workflows.

Part 1 Data Ingestion: Getting Data Into the Lakehouse

1.1 The Ingestion Problem

In a traditional database, ingestion is simple: you run INSERT INTO or use COPY to load a CSV. The database owns both the storage and the compute.

In a Lakehouse, the picture is different. Data arrives from many sources (application logs, third-party file exports, API dumps, IoT sensors) and needs to land in a governed location where any notebook, job, or dashboard can discover and access it. This is the role of Unity Catalog and Volumes.

The Core Idea

Ingestion in a Lakehouse is a two-step process: (1) land the raw file in a Unity Catalog Volume, then (2) read it into a Delta table with schema and governance. The file becomes discoverable, versioned, and queryable the moment it arrives.

1.2 Unity Catalog Volumes

A Volume is a Unity Catalog object that references a directory of files. It follows the same three-level namespace as tables:

catalog.schema.volume
└─ workspace.pipeline.raw_files
   ├── custodian_report.xlsx
   ├── holdings_v20240315.csv
   └── prices_v20240315.csv

Volumes give you the flexibility of file storage (any format, any structure) with the governance of Unity Catalog (permissions, lineage, discoverability). Think of a Volume as a governed folder: every file you upload is tracked and can be discovered by any authorized notebook or job.

1.3 Three Ways to Ingest Data

Method How It Works Best For
UI Upload Drag and drop files into a Volume via the Databricks workspace UI Small files, ad-hoc exploration, getting started
COPY INTO SQL command that loads files from a Volume into a Delta table. Idempotent (skips already-loaded files). Batch loads, repeatable ingestion
Auto Loader Streaming ingestion that watches a directory and processes new files as they arrive Continuous ingestion, production pipelines

Part 2 The Medallion Architecture: Bronze, Silver, Gold

2.1 The Problem: Raw Data Is Messy

Data arriving from external systems is rarely clean. It has missing values, inconsistent types, duplicate records, and schema drift. If you build dashboards directly on raw data, you get unreliable results. If analysts have to clean data every time they query it, you waste compute and risk inconsistency.

The Medallion Architecture (also called multi-hop architecture) solves this by organizing data into three layers, each with increasing quality and structure.

The Medallion Architecture: data flows left-to-right through layers of increasing quality.
Bronze Layer: Raw Ingestion

Capture exactly what arrived and persist it. No cleaning, no type casting, no assumptions. Add metadata (source file, timestamp) so you know when and where the data came from. If Silver logic has a bug, you can always reprocess from Bronze without re-uploading the original file.
Analogy: The inbox of an email system. Everything arrives here, unsorted.

Silver Layer: Cleaned and Conformed

Transform messy Bronze data into a single source of truth. Deduplicate records, cast types (strings become dates and integers), filter out corrupt or null rows, and standardize column names. After Silver, the data is reliable enough to build on.
Analogy: A sorted, labeled filing cabinet. Everything is organized and trustworthy.

Gold Layer: Business-Level Aggregates

Pre-aggregated, denormalized tables for specific business use cases. A Gold table might be a portfolio valuation, a daily revenue summary, or a feature table for ML. These tables are optimized for fast queries and direct consumption by dashboards and reports.
Analogy: The executive summary. Distilled, actionable, ready to present.

2.2 Why Separate Layers?

  • Decoupled processing stages. Each layer is an independent Delta table. If Silver logic has a bug, you fix it and reprocess from Bronze without re-ingesting from the source. This is the same principle as idempotent processing in message queues.
  • Incremental processing. In production, only new data flows through the pipeline. Bronze appends new files, Silver processes only unprocessed records, and Gold re-aggregates incrementally.
  • Independent scaling. Bronze ingestion can run on a high-throughput cluster. Silver cleaning can run on a schedule. Gold aggregation can be triggered by a downstream dashboard. Each stage scales independently.
  • Fault isolation. If Gold tables are corrupted by a bad aggregation, Silver is unaffected. If Silver logic breaks, Bronze preserves the raw data. Each layer acts as a checkpoint.

Part 3 The Scenario: Daily Custodian Report

You are an analyst at a small investment fund. Every day, your custodian (think State Street or BNY Mellon) sends you an Excel file with your portfolio snapshot. The file has two tabs:

Sheet What It Contains What's Messy About It
Holdings Your 50 stock positions: ticker, company name, shares, cost basis, sector Title block in rows 1-4, tickers with trailing spaces or exchange prefixes ("NYSE:MSFT"), cost values with embedded dollar signs ("$142.50"), blank rows, 3 duplicate positions
Prices End-of-day closing prices: symbol, date, close, volume, daily change Mixed date formats ("2024-03-15" vs "03/15/2024"), prices with dollar signs and commas, missing values marked as "N/A", 4 duplicate rows

Your job: build a pipeline that turns this messy Excel file into a clean, queryable portfolio valuation table showing what you own, what it's worth, and how your exposure is distributed across sectors.

Real World Context

This is a genuine scenario. Custodian reports at investment firms are notoriously messy: title blocks, inconsistent formatting, duplicate entries, and mixed date formats are the norm, not the exception. The Medallion Architecture exists precisely because of data like this.

Part 5 Run the Pipeline

5.0 Upload the Notebooks to Your Workspace

Before running the pipeline, you need to get the three notebooks into your Databricks workspace:

  1. Download the three notebook files from the course repository on GitHub: nb5_bronze.py, nb5_silver.py, and nb5_gold.py
  2. In the Databricks sidebar, click Workspace
  3. Navigate to your user folder (or create a new folder called pipeline to keep things organized)
  4. Click the ⋮ menu (three dots) and select Import
  5. Choose File and upload all three notebook files from your desktop
Upload the three pipeline notebooks to your Workspace.

With the notebooks in your workspace, run them in order. Each notebook reads from the previous layer's output and writes the next layer.

5.1 nb5_bronze.py

Reads the raw Excel file from the Volume with pandas (since Spark cannot read .xlsx natively), converts both tabs to Spark DataFrames with all-string types, adds ingestion metadata (_source_file, _source_sheet, _ingested_at), and writes two Delta tables: workspace.pipeline.bronze_holdings and workspace.pipeline.bronze_prices.

5.2 nb5_silver.py

Reads from Bronze and cleans both tables:

  • Holdings: strips whitespace from tickers, removes exchange prefixes ("NYSE:MSFT" becomes "MSFT"), removes dollar signs and commas from cost values, casts types, drops blank rows, deduplicates
  • Prices: normalizes mixed date formats, cleans dollar signs from prices, replaces "N/A" with null, casts types, deduplicates on (symbol, date)

Silver also writes versioned CSVs back to the Volume with a timestamp in the filename (e.g., holdings_v20240315_143022.csv). Navigate to the Volume in the Catalog browser to see the files accumulate across runs.

5.3 nb5_gold.py

Joins Silver holdings with Silver prices to compute a portfolio valuation table: market value per position, unrealized P&L, and portfolio weight. Also builds a sector allocation table showing exposure by sector. This is what a portfolio manager would look at every morning.

Part 6 Wiring It Together: Databricks Workflows

6.1 Why Workflows?

Running notebooks manually in order works for development, but in production you need orchestration: automatic execution, dependency management, retry on failure, and monitoring. Databricks Workflows (the Jobs UI) lets you wire your notebooks into a visual DAG that runs automatically.

The Distributed Systems Pattern

A Databricks Workflow is a DAG (Directed Acyclic Graph), the same abstraction Spark uses for query execution. Each task in the workflow is a node, and dependencies between tasks are edges. The orchestrator ensures tasks run in the correct order and handles retries on failure. This is the same pattern as AWS Step Functions, Apache Airflow, and Kubernetes Job controllers.

What You Should See

The Workflow UI shows your pipeline as a three-node graph: bronze → silver → gold. Each node turns green as it completes. This is the same DAG visualization that production data teams use to monitor pipelines processing billions of rows daily.

The Big Idea

Every distributed data system is a pipeline. The Medallion Architecture is the Lakehouse version of a pattern you have seen all semester: raw input, processing stages, refined output. Understanding this pattern means you can design, debug, and optimize data pipelines on any platform.
