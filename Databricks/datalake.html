What Is a Data Lake?

A short primer on the concept that sits underneath the Lakehouse architecture.

The Problem

Traditionally, organizations stored structured data in data warehouses — relational databases optimized for analytics (think: Snowflake, Redshift, BigQuery). But warehouses are expensive, rigid about schema, and struggle with unstructured data like logs, images, and raw JSON.

The Data Lake Idea

A data lake is a storage system that holds raw data in its native format — structured, semi-structured, and unstructured — at any scale. Instead of transforming data before loading it (ETL), you load everything first and transform it when you need it (ELT).

  • Store everything — CSV, JSON, Parquet, images, logs, video
  • Schema on read — impose structure when you query, not when you ingest
  • Cheap storage — built on object stores (S3, ADLS, GCS) at pennies per GB
  • Decouple storage from compute — many engines can read the same data

The Data Lake Problem

Without governance, data lakes became data swamps: no schema enforcement, no transactions, no way to know what data is reliable. Files got overwritten, schemas drifted, and nobody knew what was in there.

The Lakehouse Solution

The Lakehouse combines the best of both: the cheap, flexible storage of a data lake with the reliability and governance of a data warehouse. Technologies like Delta Lake add ACID transactions, schema enforcement, and time travel on top of raw file storage. Unity Catalog adds centralized governance and discovery.

This is the architecture you work with in Databricks — and the reason your tables have schema enforcement, time travel, and a three-level namespace even though they are stored as Parquet files underneath.

The One-Sentence Version

A data lake stores everything cheaply. A lakehouse makes that storage reliable. Delta Lake is the technology that bridges the gap.
