Declarative Pipelines: The Modern Approach

Instead of writing execution steps, you describe the desired state of each table. The platform handles the rest.

Part 1 The Decorator Is the Contract

1.1 What @dlt.table() Advertises

In a Declarative Pipeline, every table starts with a decorator. Think of it as the table's public interface — an advertisement of what this table is, what it promises, and what it contains. Before anyone reads a single line of your transformation logic, the decorator tells them everything they need to know:

The Contract

@dlt.table(                                       # "I am a table."
    name       = "gold_portfolio",                # "This is my name."
    comment    = "Portfolio valuation with P&L",   # "This is what I contain."
    table_properties = {"quality": "gold"},        # "This is my quality tier."
)
@dlt.expect("has_value", "Market_Value > 0")      # "I guarantee this about my data."
@dlt.expect_or_drop("valid_ticker",               # "Bad rows are removed, and I
    "Ticker IS NOT NULL")                         #  track how many were dropped."
def gold_portfolio():                             # (function name = table name)
    ...

Before you even see the transformation logic, you already know:

  • What it is — A table (not a view, not a temp scratch pad)
  • What it's called — gold_portfolio
  • What it's for — "Portfolio valuation with P&L"
  • What quality tier it belongs to — Gold
  • What data guarantees it makes — Every row has a positive Market_Value, and every row has a non-null Ticker (bad rows are dropped and counted)
The Key Idea

The decorator is like a class interface or an API specification. It declares the contract — what this table promises to the rest of the pipeline and to downstream consumers. The function body is the implementation — how that contract is fulfilled. This separation is what makes DLT declarative.

1.2 The Function Body: How the Contract Is Fulfilled

Inside the function, you write regular Spark code. The function must return a DataFrame that matches the contract declared above. Here is the complete picture:

@dlt.table(
    comment = "Portfolio valuation with P&L",
    table_properties = {"quality": "gold"},
)
@dlt.expect("has_value", "Market_Value > 0")
@dlt.expect_or_drop("valid_ticker", "Ticker IS NOT NULL")
def gold_portfolio():
    """Join holdings with prices to compute market values."""
    holdings = dlt.read("silver_holdings")       # ← "I need this table"
    prices   = dlt.read("silver_prices")         # ← "I need this table too"
    return (holdings
        .join(prices, holdings["Ticker"] == prices["Symbol"])
        .withColumn("Market_Value", col("Shares_Held") * col("Close"))
        .withColumn("Unrealized_PnL",
            col("Market_Value") - (col("Shares_Held") * col("Cost_Basis")))
    )

The dlt.read() calls inside the function serve two purposes simultaneously:

  1. They return data — You get a Spark DataFrame from that table.
  2. They declare a dependency — They tell Databricks: "this table needs those tables to exist first." Databricks uses these to build the execution order automatically.

You never write .write.saveAsTable(). You never say "run this after Silver." You just describe what the table should contain, and return a DataFrame. The platform handles persistence, ordering, retries, and tracking.

1.3 The Three Types of Declarations

Decorator What It Declares Best For
@dlt.table() "I am a streaming table." New data is appended on each run; old data is preserved. Databricks auto-determines if this is a materialized view when you return a non-streaming DataFrame. Most tables. Bronze ingestion, Silver cleaning, Gold aggregation.
@dlt.view() "I am a temporary view." I exist only during the pipeline run. I am not persisted to the catalog. Wrapping external tables, intermediate computations you don't want published.
@dlt.materialized_view() "I am a materialized view." I am fully recomputed from scratch on each run. Silver/Gold tables where you explicitly want complete-refresh semantics.
Note

In our notebook (nb6_declarative.py), we use @dlt.table() for everything. Databricks automatically determines the right semantics based on your function's return type. Full Python API reference →

Part 2 Data Quality as Part of the Contract

2.1 Expectations: Data Guarantees

In the imperative pipeline, you wrote .filter() calls to remove bad data. This works, but you have no visibility into what was filtered or how many records failed. If your filter accidentally drops valid data, you would not notice.

In a declarative pipeline, data quality rules are part of the table's contract. They are declared with @dlt.expect decorators, and Databricks tracks them in a live dashboard:

Decorator What It Declares Use When
@dlt.expect() "I want this to be true." Log violations but keep the rows. You can review them in the pipeline's data quality dashboard. Monitoring: you want to see bad data but not block the pipeline.
@dlt.expect_or_drop() "I require this." Silently drop rows that violate the rule. The dropped count is tracked. Cleaning: you know some rows will be bad and want them removed automatically.
@dlt.expect_or_fail() "I guarantee this." Stop the entire pipeline if any row violates the rule. Critical rules: data integrity must be preserved (e.g., no negative prices). 
# Example: stacking multiple expectations on one table
@dlt.table(comment="Validated holdings")
@dlt.expect("has_company", "Company_Name IS NOT NULL")           # log only
@dlt.expect_or_drop("valid_ticker", "Ticker IS NOT NULL")        # drop bad rows
@dlt.expect_or_fail("positive_cost", "Cost_Basis > 0")           # fail if violated
def silver_holdings():
    return dlt.read("bronze_holdings").filter(...)
Distributed Systems Connection

Expectations implement a pattern called circuit breaking. In microservices, a circuit breaker stops sending requests to a failing service to prevent cascading failures. @dlt.expect_or_fail() does the same thing for data: if critical data quality rules are violated, the pipeline stops before bad data pollutes downstream tables. Part 3 The Dependency Graph

3.1 Two Phases: Loading and Execution

When you click Start on a DLT pipeline, Databricks does not run your code top-to-bottom like a regular notebook. Instead, it operates in two distinct phases:

Phase 1 — Code Loading (No Data Moves Yet)

Databricks loads your notebook like a Python module. The @dlt.table() decorators fire during this phase, and each one registers its function with an internal pipeline catalog: "There is a table called bronze_holdings, and here is the function that produces it." At this point, Databricks reads the dlt.read() calls inside each function to discover the dependencies: "the function silver_holdings calls dlt.read("bronze_holdings"), so silver_holdings depends on bronze_holdings." From these dependency declarations, it builds the full DAG. No data has been read or written yet.

Phase 2 — Execution (Data Flows)

Now Databricks knows the graph. It executes the functions in topological order — Bronze first (no upstream dependencies), then Silver (dependencies satisfied), then Gold. Each function is called, produces a DataFrame, and the platform persists the result as a Delta table. Expectations are evaluated row-by-row during this phase, and violation counts are tracked in the pipeline dashboard.

# During Phase 1 (loading), Databricks discovers these edges:
#
#   bronze_holdings  ->  silver_holdings ---+
#                                          +--->  gold_portfolio  ->  gold_sector_allocation
#   bronze_prices    ->  silver_prices  ---+
#
# During Phase 2 (execution), it runs them in this order.
You defined six functions. Databricks inferred this graph from the dlt.read() calls inside them.
Important Rule

Inside DLT functions, always use dlt.read("table_name") to reference other pipeline tables. Do not use spark.read.table() or spark.table() for tables defined in the same pipeline. If you do, Databricks will not see the dependency and may run tables in the wrong order.

Part 4 What the Platform Handles for You

4.1 The Gold Table, Two Ways

To see why this approach matters, look at the exact same Gold table written both ways. The transformation logic (join, withColumn) is the same. What changes is everything around it:

Imperative (nb5_gold.py)

# You write ALL of this:
holdings = spark.read.table(
    "pipeline.silver_holdings")
prices = spark.read.table(
    "pipeline.silver_prices")
portfolio = holdings.join(prices, ...)
    .withColumn("Market_Value", ...)
portfolio.write
    .mode("overwrite")
    .saveAsTable(
        "pipeline.gold_portfolio")

# And you ALSO must:
# - Run this AFTER Silver finishes
# - Wire the order in a Workflow
# - Handle errors yourself
# - Write .filter() for bad data

Declarative (nb6)

@dlt.table(
    comment="Portfolio valuation",
    table_properties={"quality":"gold"}
)
@dlt.expect("has_value",
    "Market_Value > 0")
def gold_portfolio():
    h = dlt.read("silver_holdings")
    p = dlt.read("silver_prices")
    return h.join(p, ...)
        .withColumn("Market_Value",...)

# You did NOT write:
# ✗ No .write.saveAsTable()
# ✗ No execution order
# ✗ No Workflow wiring
# ✗ No manual error handling

4.2 Full Comparison

Aspect Imperative (Notebooks + Workflow) Declarative (DLT Pipeline)
DAG You wire tasks manually in the Workflow UI Auto-resolved from dlt.read() calls
Files One notebook per layer (3+ files) One file with all table definitions
Error handling Manual try/except or notebook failure Expectations with log/drop/fail semantics
Data quality You write filters, no visibility into drops Built-in dashboard showing pass/fail counts
Persistence You write .saveAsTable() Handled by the platform
Best for Learning, prototyping, custom logic Production pipelines, team collaboration
Both Approaches Are Valid

The imperative approach is not wrong. It gives you fine-grained control and is easier to debug step-by-step. Many teams start with imperative notebooks and migrate to declarative pipelines as their pipelines mature. Understanding both approaches is essential.

Part 5 @dlt.table() Parameter Reference

Parameter Type What It Declares
name str "My name is X." If omitted, uses the function name.
comment str "I contain X." A human-readable description, visible in the Catalog UI.
table_properties dict "These are my properties." E.g., {"quality": "silver"} for tagging, or {"delta.autoOptimize.optimizeWrite": "true"}.
schema StructType / str "My columns look like this." An explicit schema (PySpark StructType or SQL DDL string).
partition_cols list "Partition me by these columns." Useful for large tables queried by date or region.
cluster_by list "Cluster me by these columns." Uses liquid clustering, the modern alternative to partitioning.
spark_conf dict "Use these Spark settings for my query only." Scoped to this table's execution.
private bool "I am internal." If True, the table is not published to the metastore.

6.4 Design Pattern: How to Think Declaratively

When building a new pipeline, follow this process:

  1. Start with the end. What Gold tables do you need? Define those functions first.
  2. Identify dependencies. What does each Gold table need? Those are your Silver tables.
  3. Trace to sources. What raw data feeds Silver? Those are your Bronze tables.
  4. Add quality rules. At each layer, what must be true? Add expectations.

Each function only references its closest upstream step. You never skip layers. If you add a new table later, you just define a new function that reads from an existing table, and the DAG updates automatically.

The Big Idea

Declarative pipelines are the data engineering equivalent of Infrastructure as Code. Terraform declares "I want this server," not "SSH in, install packages, configure the firewall." DLT declares "I want this table, with these contents and these quality rules." The platform handles the how.
